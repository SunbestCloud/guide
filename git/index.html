Core Concepts

The building blocks of every GitLab pipeline

GitLab CI/CD is driven by a single file — .gitlab-ci.yml — in your repo root. Here's how all the pieces fit together.

🔀 Pipeline

The full automation run triggered by an event — push, MR, or schedule. Contains all stages and jobs.

📦 Stage

A phase of the pipeline like build → test → deploy. All jobs in a stage run in parallel.

⚙️ Job

A single task within a stage. Each job runs in a fresh container and executes your script.

🤖 Runner

The agent that executes jobs. Can be GitLab's shared runners or your own self-hosted runner.

📁 Artifact

Files produced by a job and passed to later jobs or downloaded. Built binaries, test reports, etc.

💾 Cache

Reused dependencies across pipeline runs to speed things up. node_modules, pip packages, etc.

💡
Key Distinction

Use artifacts to pass files between jobs in the same pipeline. Use cache to speed up repeated runs by reusing downloaded dependencies.

The complete picture — every directive you need to know

Basic Structure

.gitlab-ci.yml YAML 
stages:
  - build
  - test
  - deploy

variables:
  APP_NAME: my-app

build-job:
  stage: build
  script:
    - echo "Building $APP_NAME"
    - npm install
    - npm run build

test-job:
  stage: test
  script:
    - npm run test

deploy-job:
  stage: deploy
  script:
    - echo "Deploying..."
  only:
    - main

Rules — Preferred over only/except

rules example YAML 
deploy-job:
  script: ./deploy.sh
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'   # only on main
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
      when: never                          # skip on MRs

needs — DAG (run out of stage order)

needs + parallel YAML 
deploy-job:
  stage: deploy
  needs: ["build-job"]  # starts as soon as build-job finishes

test-job:
  parallel: 5           # runs 5 instances simultaneously
  script: npm run test -- --shard=$CI_NODE_INDEX/$CI_NODE_TOTAL

Docker in GitLab CI/CD

Building, pushing, and running containers in your pipeline

🖥️ Docker as Runtime

Use a Docker image as the environment your job runs in. Every job can specify its own image:.

🔨 Docker as Task

Build, tag, and push Docker images from your pipeline. Requires Docker-in-Docker (DinD).

Docker-in-Docker Setup

build and push to GitLab Registry YAML 
build-and-push:
  image: docker:26
  services:
    - docker:26-dind            # sidecar Docker daemon
  variables:
    DOCKER_TLS_CERTDIR: "/certs"
    IMAGE_TAG: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
  before_script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
  script:
    - docker build
        --cache-from $CI_REGISTRY_IMAGE:latest
        --tag $IMAGE_TAG
        --tag $CI_REGISTRY_IMAGE:latest .
    - docker push $IMAGE_TAG
    - docker push $CI_REGISTRY_IMAGE:latest
VariableValueSource
$CI_REGISTRYregistry.gitlab.comAuto-provided
$CI_REGISTRY_IMAGEregistry.gitlab.com/group/projectAuto-provided
$CI_REGISTRY_USERGitLab CI token userAuto-provided

🔒
Security Best Practices

Never hardcode credentials. Use $CI_REGISTRY_USER for GitLab's built-in registry. For Docker Hub, store tokens in Settings → CI/CD → Variables with the Masked option enabled.

Terraform Explained

Infrastructure as code — the complete mental model

Think of Terraform as a universal remote control for cloud infrastructure. Instead of clicking around the AWS console, you write code that describes what you want — and Terraform figures out how to make it real.

The 3-Step Cycle

1. terraform init

Downloads the AWS provider plugin that knows how to talk to AWS APIs. Sets up the remote state backend.

2. terraform plan

Compares your code vs what currently exists. Shows exactly what will be created, changed, or destroyed. Nothing touches AWS yet.

3. terraform apply

Executes the plan, calls AWS APIs, waits for resources to be ready. Type yes to confirm.

The 4 Core Concepts

core HCL concepts HCL 
# 1. RESOURCE — things you create
resource "aws_s3_bucket" "my_bucket" {
  bucket = "my-unique-bucket-2024"
}

# 2. VARIABLE — inputs to your config
variable "environment" {
  type    = string
  default = "dev"
}

# 3. OUTPUT — values you want to see or share
output "bucket_url" {
  value = aws_s3_bucket.my_bucket.bucket_domain_name
}

# 4. DATA SOURCE — read existing things you didn't create
data "aws_vpc" "existing" {
  tags = { Name = "company-main-vpc" }
}

resource "aws_subnet" "mine" {
  vpc_id     = data.aws_vpc.existing.id  # reference it with data.
  cidr_block = "10.0.5.0/24"
}

Commands Cheat Sheet

terminal BASH 
terraform init          # download providers, set up backend
terraform validate      # check syntax without connecting to AWS
terraform fmt           # auto-format your .tf files
terraform plan          # show what will change (dry run)
terraform apply         # make it real
terraform destroy       # tear everything down
terraform output        # show output values
terraform state list    # list all resources in state
terraform import        # bring existing AWS resource under Terraform
🚨
Common Gotchas

Never edit state manually — use terraform state commands. terraform destroy destroys everything — always run plan first. Sensitive values end up in state — use SSM Parameter Store for passwords, not raw variables.

Infrastructure Walkthrough

VPC → ECR → ALB → ECS Fargate — explained line by line

1

VPC & Networking

Creates the isolated network. Public subnets for the ALB (internet-facing), private subnets for ECS tasks (no direct internet access). NAT Gateways let private tasks pull Docker images from ECR without being publicly reachable.

2

Security Groups

ALB security group accepts port 80/443 from the internet. ECS security group only accepts traffic from the ALB security group — not from an IP range, but from a specific AWS identity. Nothing else can reach your containers.

3

ECR Repository

Private container registry. Auto-scans images on push for vulnerabilities. Lifecycle policy keeps last 10 images and deletes older ones to control storage costs.

4

ALB + Target Group

Load balancer spreads traffic across all ECS tasks. Target type must be ip for Fargate. Health checks hit /health every 30s — tasks failing 3 checks are removed from rotation automatically.

5

ECS Cluster + Fargate Service

Cluster is a logical namespace. Service maintains desired_count tasks always running. deployment_circuit_breaker auto-rolls back if new tasks fail health checks. ignore_changes = [task_definition] lets CI/CD manage image updates without Terraform fighting over it.

6

Auto Scaling

Watches average CPU across all tasks. Scales out quickly (60s cooldown) when CPU hits 70%. Scales in slowly (300s cooldown) to avoid yo-yoing. Min = your desired count. Max = 6 tasks (cost protection).

run it BASH 
terraform init       # downloads provider, connects to S3 backend
terraform validate   # check syntax
terraform fmt        # auto-format
terraform plan       # preview ~27 resources (always do this first)
terraform apply      # create everything (~3-5 min)
terraform output     # see your ALB URL + ECR push commands

# Tear down to avoid AWS costs
terraform destroy

🟢 Project 1 — Number Guessing Game

Concepts: variables, loops, conditionals, user input, random

guess.py PYTHON 
import random

def play_game():
    secret = random.randint(1, 100)
    attempts = 0
    max_attempts = 7

    print("🎯 Guess the number between 1 and 100!")

    while attempts < max_attempts:
        try:
            guess = int(input(f"Attempt {attempts+1}/{max_attempts} → "))
        except ValueError:
            print("Please enter a valid number.")
            continue

        attempts += 1
        if guess < secret:
            print("📈 Too low!")
        elif guess > secret:
            print("📉 Too high!")
        else:
            print(f"🎉 Correct in {attempts} attempts!")
            return

    print(f"💀 Game over! The number was {secret}.")

if __name__ == "__main__":
    play_game()

🟢 Project 2 — Expense Tracker

Concepts: functions, lists, dictionaries, file I/O, JSON

expenses.py PYTHON 
import json, os
from datetime import datetime

FILENAME = "expenses.json"

def load_expenses():
    if os.path.exists(FILENAME):
        with open(FILENAME) as f:
            return json.load(f)
    return []

def add_expense(expenses):
    category    = input("Category: ").strip()
    description = input("Description: ").strip()
    amount      = float(input("Amount: $"))

    expenses.append({
        "date": datetime.now().strftime("%Y-%m-%d"),
        "category": category,
        "description": description,
        "amount": amount
    })
    with open(FILENAME, "w") as f:
        json.dump(expenses, f, indent=2)

def show_summary(expenses):
    total = sum(e["amount"] for e in expenses)
    by_cat = {}
    for e in expenses:
        by_cat[e["category"]] = by_cat.get(e["category"], 0) + e["amount"]
    print(f"💰 Total: ${total:.2f}")
    for cat, amt in sorted(by_cat.items(), key=lambda x: x[1], reverse=True):
        bar = "█" * int(amt / total * 20)
        print(f"  {cat:<15} ${amt:>8.2f}  {bar}")

🟢 Project 3 — Weather CLI

Concepts: HTTP requests, APIs, error handling, environment variables

weather.py PYTHON 
import os, sys, requests

API_KEY  = os.getenv("OPENWEATHER_API_KEY")
BASE_URL = "https://api.openweathermap.org/data/2.5"

def get_weather(city):
    r = requests.get(f"{BASE_URL}/weather",
        params={"q": city, "appid": API_KEY, "units": "metric"},
        timeout=10)
    r.raise_for_status()
    return r.json()

def display_weather(data):
    temp  = data["main"]["temp"]
    desc  = data["weather"][0]["description"].capitalize()
    humid = data["main"]["humidity"]
    wind  = data["wind"]["speed"]
    print(f"🌡️  {data['name']}: {temp:.1f}°C — {desc}")
    print(f"   Humidity: {humid}%  Wind: {wind} m/s")

if __name__ == "__main__":
    city = " ".join(sys.argv[1:]) or input("City: ")
    try:
        display_weather(get_weather(city))
    except requests.HTTPError as e:
        print(f"❌ {e}")

🟡 Project 4 — FastAPI Task Manager

Concepts: REST API, FastAPI, Pydantic validation, HTTP methods, auto-generated docs

# models.py
from pydantic import BaseModel, Field
from enum import Enum
from typing import Optional
from datetime import datetime

class Priority(str, Enum):
    low    = "low"
    medium = "medium"
    high   = "high"

class TaskCreate(BaseModel):
    title       : str = Field(..., min_length=1, max_length=100)
    description : Optional[str] = None
    priority    : Priority = Priority.medium

# routers/tasks.py
from fastapi import APIRouter, HTTPException

router = APIRouter(prefix="/tasks", tags=["tasks"])
db = {}; counter = 0

@router.post("/", status_code=201)
def create_task(task: TaskCreate):
    global counter; counter += 1
    db[counter] = {"id": counter, **task.model_dump()}
    return db[counter]

@router.get("/{task_id}")
def get_task(task_id: int):
    if task_id not in db:
        raise HTTPException(status_code=404, detail="Not found")
    return db[task_id]

🟡 Project 5 — PostgreSQL with SQLAlchemy Async

Concepts: async SQLAlchemy ORM, Alembic migrations, dependency injection

from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession
from sqlalchemy.orm import sessionmaker, DeclarativeBase
import os

DATABASE_URL = os.getenv("DATABASE_URL",
    "postgresql+asyncpg://postgres:password@localhost/taskdb")

engine = create_async_engine(DATABASE_URL)

AsyncSessionLocal = sessionmaker(
    engine, class_=AsyncSession, expire_on_commit=False)

class Base(DeclarativeBase): pass

async def get_db():
    async with AsyncSessionLocal() as session:
        try:
            yield session
            await session.commit()
        except:
            await session.rollback()
            raise
