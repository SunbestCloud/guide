Ultra-Compact GitHub Actions CI/CD

File: .github/workflows/ci-cd.yml

name: CI/CD
on:
  push: { branches: [main] }
  pull_request: { branches: [main] }
jobs:
  pipeline:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: npm }
      - run: npm ci
      - run: npm test
      - uses: aquasecurity/trivy-action@master
        with: { scan-type: fs, severity: HIGH,CRITICAL }
      - run: docker build -t app:${{ github.sha }} .
      - run: trivy image --severity HIGH,CRITICAL --exit-code 1 app:${{ github.sha }}
      - uses: docker/login-action@v3
        if: github.event_name == 'push'
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}
      - run: |
          docker tag app:${{ github.sha }} ${{ secrets.DOCKERHUB_USERNAME }}/my-app:${{ github.sha }}
          docker tag app:${{ github.sha }} ${{ secrets.DOCKERHUB_USERNAME }}/my-app:latest
          docker push ${{ secrets.DOCKERHUB_USERNAME }}/my-app:${{ github.sha }}
          docker push ${{ secrets.DOCKERHUB_USERNAME }}/my-app:latest
        if: github.event_name == 'push'
      - run: |
          curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64
          sudo install minikube-linux-amd64 /usr/local/bin/minikube
          minikube start --driver=docker
        if: github.event_name == 'push'
      - run: |
          kubectl apply -f k8s/deployment.yaml
          kubectl set image deployment/my-app my-app=${{ secrets.DOCKERHUB_USERNAME }}/my-app:${{ github.sha }}
          kubectl rollout status deployment/my-app --timeout=120s
          curl -f $(minikube service my-app --url)/ || exit 1
        if: github.event_name == 'push'

Required Secrets: DOCKERHUB_USERNAME, DOCKERHUB_TOKEN

Setting Up Linux Servers for Your DevOps Lab

January 04, 2026

1. Base OS Recommendation

Use Ubuntu 24.04 LTS (Noble Numbat) — stable, well-supported, excellent for DevOps tools.

2. Local Setup (Virtual Machine)

  1. Download Ubuntu 24.04 LTS ISO from ubuntu.com
  2. Use VirtualBox, VMware, or Hyper-V
  3. Recommended specs: 4+ CPUs, 8+ GB RAM, 50+ GB disk
  4. During install: Enable OpenSSH server

3. Initial Server Hardening & Updates

sudo apt update && sudo apt upgrade -y
sudo apt install -y curl wget gnupg ca-certificates unzip git
# Enable automatic security updates
sudo apt install -y unattended-upgrades
sudo dpkg-reconfigure --priority=low unattended-upgrades
# Basic firewall
sudo ufw allow OpenSSH
sudo ufw enable

4. Create a Non-Root User (Best Practice)

sudo adduser devopsuser
sudo usermod -aG sudo devopsuser
# Copy SSH keys
sudo mkdir -p /home/devopsuser/.ssh
sudo cp ~/.ssh/authorized_keys /home/devopsuser/.ssh/
sudo chown -R devopsuser:devopsuser /home/devopsuser/.ssh
# Disable root SSH login (edit /etc/ssh/sshd_config)
sudo sed -i 's/PermitRootLogin yes/PermitRootLogin no/' /etc/ssh/sshd_config
sudo systemctl restart ssh

5. Cloud Server Setup (AWS EC2 Example)

  1. Launch EC2 instance → Ubuntu Server 24.04 LTS
  2. Instance type: t3.medium or larger
  3. Security Group: Allow SSH (22) from your IP
  4. Key pair: Create/download .pem
  5. Connect: ssh -i key.pem ubuntu@ec2-public-ip

More Ansible Playbook Examples

January 04, 2026 — Practical, modular playbooks for common DevOps tasks

1. Playbook: Install and Configure Prometheus Node Exporter

File: playbook-monitoring.yml

---
- name: Install Prometheus Node Exporter
  hosts: lab_servers
  become: yes
  vars:
    node_exporter_version: "1.8.2"

  tasks:
    - name: Create node_exporter user
      user:
        name: node_exporter
        system: yes
        shell: /usr/sbin/nologin

    - name: Download and extract Node Exporter
      unarchive:
        src: "https://github.com/prometheus/node_exporter/releases/download/v{{ node_exporter_version }}/node_exporter-{{ node_exporter_version }}.linux-amd64.tar.gz"
        dest: /tmp
        remote_src: yes

    - name: Move binary
      copy:
        src: "/tmp/node_exporter-{{ node_exporter_version }}.linux-amd64/node_exporter"
        dest: /usr/local/bin/node_exporter
        owner: node_exporter
        group: node_exporter
        mode: '0755'
        remote_src: yes

    - name: Create systemd service
      copy:
        dest: /etc/systemd/system/node_exporter.service
        content: |
          [Unit]
          Description=Prometheus Node Exporter

          [Service]
          User=node_exporter
          ExecStart=/usr/local/bin/node_exporter
          Restart=always

          [Install]
          WantedBy=multi-user.target

    - name: Enable and start service
      systemd:
        name: node_exporter
        enabled: yes
        state: started

    - name: Cleanup
      file:
        path: "/tmp/node_exporter-{{ node_exporter_version }}.linux-amd64"
        state: absent

Run: ansible-playbook playbook-monitoring.yml
Metrics available at http://server-ip:9100/metrics

2. Playbook: Deploy Dockerized Node.js App

File: playbook-deploy-app.yml

---
- name: Deploy Node.js App via Docker
  hosts: lab_servers
  become: yes
  vars:
    app_image: "{{ dockerhub_username }}/my-app:latest"
    app_port: 3000

  tasks:
    - name: Pull latest image
      docker_image:
        name: "{{ app_image }}"
        source: pull

    - name: Remove old container
      docker_container:
        name: my-app
        state: absent
      ignore_errors: yes

    - name: Run new container
      docker_container:
        name: my-app
        image: "{{ app_image }}"
        state: started
        restart_policy: always
        published_ports:
          - "{{ app_port }}:3000"

    - name: Wait for app to be healthy
      uri:
        url: "http://localhost:{{ app_port }}"
        return_content: yes
      register: result
      until: result.status == 200
      retries: 10
      delay: 5

Run: ansible-playbook playbook-deploy-app.yml -e "dockerhub_username=youruser"

3. Playbook: Full Security Hardening

File: playbook-harden.yml

---
- name: Harden Ubuntu Servers
  hosts: lab_servers
  become: yes
  tasks:
    - name: Upgrade all packages
      apt:
        upgrade: dist
        update_cache: yes

    - name: Enable automatic security updates
      apt:
        name: unattended-upgrades
        state: present

    - name: Configure unattended-upgrades
      copy:
        dest: /etc/apt/apt.conf.d/50unattended-upgrades
        content: |
          Unattended-Upgrade::Allowed-Origins {
              "${distro_id}:${distro_codename}-security";
          };

    - name: Disable password authentication
      lineinfile:
        path: /etc/ssh/sshd_config
        regexp: '^PasswordAuthentication'
        line: 'PasswordAuthentication no'
      notify: restart ssh

    - name: Set idle timeout
      lineinfile:
        path: /etc/profile
        line: 'TMOUT=900 ; export TMOUT'
        create: yes

    - name: Enable and configure firewall
      block:
        - ufw:
            state: enabled
            policy: deny
        - ufw:
            rule: allow
            name: OpenSSH
        - ufw:
            rule: allow
            port: "{{ item }}"
          loop: [3000, 9090, 3000, 9100]  # app, Prometheus, Grafana, Node Exporter

  handlers:
    - name: restart ssh
      service:
        name: ssh
        state: restarted

- name: Backup Critical Configs

  become: yes
  tasks:
    - name: Create backup directory
      file:
        path: /backups/{{ inventory_hostname }}
        state: directory
        mode: '0700'

    - name: Backup configs
      copy:
        src: "{{ item }}"
        dest: "/backups/{{ inventory_hostname }}/{{ item | basename }}"
        remote_src: yes
      loop:
        - /etc/ssh/sshd_config
        - /etc/docker/daemon.json
        - /etc/prometheus/prometheus.yml

    - name: Fetch backups locally
      fetch:
        src: "/backups/{{ inventory_hostname }}/{{ item | basename }}"
        dest: "local_backups/{{ inventory_hostname }}/"
        flat: yes
      loop:
        - /etc/ssh/sshd_config
        - /etc/docker/daemon.json

Ansible Role: Prometheus Monitoring

January 04, 2026 — Install and configure Prometheus Node Exporter + optional Prometheus server

Role Structure

ansible/roles/prometheus/
├── defaults/main.yml
├── tasks/main.yml
├── templates/prometheus.yml.j2      # Optional for full server
└── handlers/main.yml

1. defaults/main.yml

prometheus_node_exporter_version: "1.8.2"
prometheus_node_exporter_enabled: true

# Optional: Install full Prometheus server
prometheus_server_enabled: false
prometheus_version: "2.53.1"
prometheus_scrape_interval: "15s"
prometheus_extra_scrape_configs: []

2. tasks/main.yml

---
- name: Install Prometheus Node Exporter
  when: prometheus_node_exporter_enabled
  block:
    - name: Create prometheus user
      user:
        name: prometheus
        system: yes
        shell: /usr/sbin/nologin

    - name: Download Node Exporter
      get_url:
        url: "https://github.com/prometheus/node_exporter/releases/download/v{{ prometheus_node_exporter_version }}/node_exporter-{{ prometheus_node_exporter_version }}.linux-amd64.tar.gz"
        dest: /tmp/node_exporter.tar.gz

    - name: Extract Node Exporter
      unarchive:
        src: /tmp/node_exporter.tar.gz
        dest: /tmp
        remote_src: yes

    - name: Install binary
      copy:
        src: "/tmp/node_exporter-{{ prometheus_node_exporter_version }}.linux-amd64/node_exporter"
        dest: /usr/local/bin/node_exporter
        owner: prometheus
        group: prometheus
        mode: '0755'
        remote_src: yes

    - name: Create systemd service
      template:
        src: node_exporter.service.j2
        dest: /etc/systemd/system/node_exporter.service
      notify: restart node_exporter

    - name: Enable and start Node Exporter
      systemd:
        name: node_exporter
        enabled: yes
        state: started
        daemon_reload: yes

    - name: Cleanup
      file:
        path: "{{ item }}"
        state: absent
      loop:
        - /tmp/node_exporter.tar.gz
        - "/tmp/node_exporter-{{ prometheus_node_exporter_version }}.linux-amd64"

- name: Install Full Prometheus Server (optional)
  when: prometheus_server_enabled
  block:
    - name: Create config directory
      file:
        path: /etc/prometheus
        state: directory
        owner: prometheus
        group: prometheus

    - name: Download Prometheus
      get_url:
        url: "https://github.com/prometheus/prometheus/releases/download/v{{ prometheus_version }}/prometheus-{{ prometheus_version }}.linux-amd64.tar.gz"
        dest: /tmp/prometheus.tar.gz

    - name: Extract and install
      unarchive:
        src: /tmp/prometheus.tar.gz
        dest: /opt
        remote_src: yes

    - name: Link binaries
      file:
        src: "/opt/prometheus-{{ prometheus_version }}.linux-amd64/{{ item }}"
        dest: "/usr/local/bin/{{ item }}"
        state: link
      loop:
        - prometheus
        - promtool

    - name: Render config
      template:
        src: prometheus.yml.j2
        dest: /etc/prometheus/prometheus.yml
        owner: prometheus
        group: prometheus
      notify: restart prometheus

    - name: Create systemd service for Prometheus
      template:
        src: prometheus.service.j2
        dest: /etc/systemd/system/prometheus.service
      notify: restart prometheus

    - name: Enable and start Prometheus
      systemd:
        name: prometheus
        enabled: yes
        state: started
        daemon_reload: yes

2. Ansible: Use Dynamic Inventory from Terraform

Option A: JSON Inventory File (Simple)

Option B: Dynamic Inventory Script (Recommended)

File: ansible/inventory/terraform_inventory.py

#!/usr/bin/env python3
import json
import subprocess
import sys

try:
    output = subprocess.check_output(["terraform", "output", "-json", "ansible_inventory"])
    inventory = json.loads(output)
    print(json.dumps(inventory))
except:
    print(json.dumps({"_meta": {"hostvars": {}}}))
    sys.exit(0)

Make executable: chmod +x terraform_inventory.py

In ansible.cfg:

[defaults]
inventory = ./inventory/terraform_inventory.py

Now just run:

ansible-playbook playbook-server-setup.yml

Ansible Role: Minikube Setup

January 04, 2026 — Fully automated single-node Kubernetes cluster with Minikube using Docker driver

Role Structure

ansible/roles/minikube/
├── defaults/main.yml
├── tasks/main.yml
└── handlers/main.yml

1. defaults/main.yml

minikube_version: "latest"                 # or specific e.g. "v1.33.1"
minikube_driver: "docker"                  # Recommended for headless servers
minikube_cpus: 4
minikube_memory: "8192mb"
minikube_disk: "50g"
minikube_addons:
  - dashboard
  - metrics-server
  - ingress

2. tasks/main.yml

---
- name: Install Minikube binary
  get_url:
    url: >-
      {% if minikube_version == "latest" %}
      https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64
      {% else %}
      https://storage.googleapis.com/minikube/releases/{{ minikube_version }}/minikube-linux-amd64
      {% endif %}
    dest: /usr/local/bin/minikube
    mode: '0755'
    owner: root
    group: root

- name: Install kubectl
  shell: |
    curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
    install kubectl /usr/local/bin/kubectl
  args:
    creates: /usr/local/bin/kubectl

- name: Ensure docker group exists and user is member
  user:
    name: "{{ ansible_user }}"
    groups: docker
    append: yes

- name: Start Minikube cluster
  become: no                                       # Run as non-root user
  command: >
    minikube start
    --driver={{ minikube_driver }}
    --cpus={{ minikube_cpus }}
    --memory={{ minikube_memory }}
    --disk-size={{ minikube_disk }}
    --force                                        # Use if non-root Docker
  environment:
    HOME: /home/{{ ansible_user }}
  register: minikube_start
  changed_when: "'Starting' in minikube_start.stdout or 'Restarting' in minikube_start.stdout"

- name: Wait for cluster to be ready
  become: no
  command: minikube status --format="{{ '{{' }} .Node.Status {{ '}}' }}"
  register: minikube_status
  until: minikube_status.stdout == "Running"
  retries: 20
  delay: 10

- name: Enable Minikube addons
  become: no
  command: minikube addons enable {{ item }}
  loop: "{{ minikube_addons }}"
  when: minikube_addons is defined

3. handlers/main.yml (optional)

---
- name: Restart Minikube
  become: no
  command: minikube stop && minikube start

Ansible Role: Helm Installation

January 04, 2026 — Install and configure Helm (Kubernetes package manager) on Minikube-enabled servers

Role Structure

ansible/roles/helm/
├── defaults/main.yml
├── tasks/main.yml
└── handlers/main.yml (optional)

1. defaults/main.yml

helm_version: "v3.14.0"                    # Check latest at https://github.com/helm/helm/releases
helm_install_dir: "/usr/local/bin"
helm_binary_name: "helm"
helm_repositories:
  - name: stable
    url: https://charts.helm.sh/stable
  - name: bitnami
    url: https://charts.bitnami.com/bitnami
  - name: ingress-nginx
    url: https://kubernetes.github.io/ingress-nginx
helm_plugins: []                           # e.g. ["helm-secrets", "helm-diff"]

2. tasks/main.yml

---
- name: Download Helm binary
  get_url:
    url: "https://get.helm.sh/helm-{{ helm_version }}-linux-amd64.tar.gz"
    dest: /tmp/helm.tar.gz
    mode: '0644'

- name: Extract Helm archive
  unarchive:
    src: /tmp/helm.tar.gz
    dest: /tmp
    remote_src: yes

- name: Install Helm binary
  copy:
    src: "/tmp/linux-amd64/helm"
    dest: "{{ helm_install_dir }}/{{ helm_binary_name }}"
    mode: '0755'
    remote_src: yes

- name: Cleanup temporary files
  file:
    path: "{{ item }}"
    state: absent
  loop:
    - /tmp/helm.tar.gz
    - /tmp/linux-amd64

- name: Verify Helm installation
  command: helm version --short
  register: helm_version_check
  changed_when: false

- name: Add Helm repositories
  command: helm repo add {{ item.name }} {{ item.url }}
  loop: "{{ helm_repositories }}"
  when: helm_repositories is defined
  changed_when: true

- name: Update Helm repo cache
  command: helm repo update
  when: helm_repositories is defined

- name: Install Helm plugins (optional)
  command: helm plugin install {{ item }}
  loop: "{{ helm_plugins }}"
  when: helm_plugins is defined and helm_plugins | length > 0
  ignore_errors: yes   # Some plugins may already be installed

3. Use in Playbook

---
- name: Install Helm on Kubernetes Nodes
  hosts: lab_servers
  become: yes
  pre_tasks:
    - name: Ensure Minikube is running (prerequisite)
      include_role:
        name: minikube

  roles:
    - role: helm
      helm_version: "v3.14.0"
      helm_repositories:
        - name: bitnami
          url: https://charts.bitnami.com/bitnami
        - name: prometheus-community
          url: https://prometheus-community.github.io/helm-charts
        - name: grafana
          url: https://grafana.github.io/helm-charts

EKS IAM Roles for Service Accounts (IRSA)

January 04, 2026 — Secure pod access to AWS resources using Terraform

What is IRSA?

IAM Roles for Service Accounts (IRSA) allows Kubernetes pods to assume IAM roles via OpenID Connect (OIDC). This provides fine-grained, temporary AWS permissions without storing credentials in pods.

1. Enable IRSA in EKS Module

In your EKS module (terraform-aws-modules/eks/aws ~> 20.0+):

module "eks" {
  source  = "terraform-aws-modules/eks/aws"
  version = "~> 20.0"

  cluster_name = var.cluster_name
  # ... other config ...

  enable_irsa = true   # Creates OIDC provider automatically
}

2. Create IRSA Role (Using Dedicated IAM Module)

Best practice: Use terraform-aws-modules/iam/aws//modules/iam-role-for-service-accounts-eks

module "my_app_irsa" {
  source  = "terraform-aws-modules/iam/aws//modules/iam-role-for-service-accounts-eks"
  version = "~> 5.0"

  role_name = "my-app-s3-access"

  # Attach managed or custom policies
  attach_s3_policy = true
  s3_bucket_arns   = ["arn:aws:s3:::my-app-bucket/*"]

  # Or custom policy
  # role_policy_arns = [aws_iam_policy.my_custom.arn]

  oidc_providers = {
    main = {
      provider_arn               = module.eks.oidc_provider_arn
      namespace_service_accounts = ["my-namespace:my-service-account"]
    }
  }

  tags = {
    Purpose = "my-app-s3-access"
  }
}

Output role ARN: module.my_app_irsa.iam_role_arn

3. Kubernetes Service Account (Apply via kubectl or Helm)

apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-service-account
  namespace: my-namespace
  annotations:
    eks.amazonaws.com/role-arn: ${module.my_app_irsa.iam_role_arn}

4. Use in Deployment

spec:
  serviceAccountName: my-service-account
  containers:
    - name: my-app
      image: my-app:latest
      # No AWS credentials needed!

5. Common IRSA Examples

Use Case Policy Attachment Service Account
EBS CSI Driver attach_ebs_csi_policy = true kube-system:ebs-csi-controller-sa
Load Balancer Controller attach_load_balancer_controller_policy = true kube-system:aws-load-balancer-controller
Cluster Autoscaler attach_cluster_autoscaler_policy = true kube-system:cluster-autoscaler
S3 Access (Custom) attach_s3_policy = true + bucket ARNs your-ns:your-sa

6. Verification

# Check pod assumes role
kubectl exec -it <pod> -- env | grep AWS_ROLE_ARN

# Test AWS access (example S3)
kubectl exec -it <pod> -- aws s3 ls my-app-bucket

7. Best Practices (2026)

  • Always enable IRSA in EKS module
  • Use dedicated IAM module for roles
  • Least privilege policies
  • Separate namespaces per app
  • Avoid node instance profiles for app permissions
