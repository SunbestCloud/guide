Core Concepts

GitLab CI/CD is driven by a single file: .gitlab-ci.yml in your repo root. When you push code, GitLab reads this file and runs your pipeline automatically.

ConceptWhat it isAnalogy
PipelineThe full automation run triggered by an event (push, MR, schedule)An assembly line
StageA phase of the pipeline (build → test → deploy)A station on the line
JobA single task within a stageOne worker at a station
RunnerThe agent that executes jobsThe worker themselves
ArtifactFiles passed between jobs/stagesA box on a conveyor belt
CacheDependencies stored to speed up future runsA toolbox that persists
EnvironmentA named deployment target (staging, production)A delivery address

Writing .gitlab-ci.yml

The pipeline file defines everything. Here's a complete example with the most important directives you'll use daily.

YAML .gitlab-ci.yml 
# Define pipeline stages in order
stages:
  - build
  - test
  - deploy

variables:
  APP_NAME: my-app
  NODE_ENV: test

# ── Build job ────────────────────────────
build-job:
  stage: build
  image: node:20-alpine
  cache:
    key:
      files: [package-lock.json]
    paths: [node_modules/]
  script:
    - npm ci
    - npm run build
  artifacts:
    paths: [dist/]
    expire_in: 1 day

# ── Test job (runs parallel to lint) ─────
unit-tests:
  stage: test
  image: node:20-alpine
  needs: [build-job]   # DAG: start as soon as build finishes
  script:
    - npm run test
  artifacts:
    reports:
      junit: junit.xml

# ── Deploy (only on main branch) ─────────
deploy-job:
  stage: deploy
  needs: [unit-tests]
  script:
    - echo "Deploying $APP_NAME..."
    - ./scripts/deploy.sh
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
      when: manual   # human approval required

Key Directives

🔀
rules (vs only/except)
Preferred modern approach. Conditionally include/exclude jobs based on branch, source, variables. More powerful than only/except.
⚡
needs (DAG)
Run jobs out of stage order. A job with needs starts the moment its dependencies finish — doesn't wait for the whole stage.
🔢
parallel
Fan out a single job into N simultaneous instances. Use $CI_NODE_INDEX and $CI_NODE_TOTAL to shard test suites.
📦
extends
Clean reuse of job configuration. Cleaner than YAML anchors. Inherit image, before_script, cache, and more.
📁
include
Split large pipelines across files. Include local files, files from other projects, or official GitLab templates.
🏃
before_script
Commands that run before every script in a job. Great for auth setup, installing dependencies, configuring environments.

Artifacts & Caching

💡
Key rule: Use cache for dependencies (node_modules, pip packages). Use artifacts for build outputs (dist/, binaries, test reports).
YAML Artifacts + Cache example 
build-job:
  cache:
    key:
      files: [package-lock.json]   # bust cache when lockfile changes
    paths: [node_modules/]
  script:
    - npm ci
    - npm run build
  artifacts:
    paths:
      - dist/              # passed to downstream jobs
    expire_in: 1 week    # auto cleanup
    reports:
      junit: test-results.xml          # shows in MR UI
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura.xml

test-job:
  script:
    - ls dist/    # dist/ auto-downloaded from build-job artifact

# Per-branch cache isolation
build-isolated:
  cache:
    key: "$CI_COMMIT_REF_SLUG"   # separate cache per branch
    paths: [.npm/]
  script:
    - npm ci --cache .npm/

Docker as runtimeUse a Docker image as the environment your job runs inimage: node:20-alpine

build-image:
  image: docker:26            # gives you the docker CLI

    DOCKER_TLS_CERTDIR: "/certs"
  script:

    - docker images

Push to Amazon ECR

YAML Build + push to ECR 
build-and-push-ecr:
  image: docker:26
  services:
    - docker:26-dind
  variables:
    DOCKER_TLS_CERTDIR: "/certs"
    ECR_REGISTRY: 123456789.dkr.ecr.us-east-1.amazonaws.com
    IMAGE_TAG: $CI_COMMIT_SHORT_SHA
  before_script:
    - apk add --no-cache aws-cli
    - aws ecr get-login-password --region us-east-1
        | docker login --username AWS --password-stdin $ECR_REGISTRY
  script:
    - docker pull $ECR_REGISTRY/my-app:latest || true
    - docker build
        --cache-from $ECR_REGISTRY/my-app:latest
        --tag $ECR_REGISTRY/my-app:$IMAGE_TAG
        --tag $ECR_REGISTRY/my-app:latest .
    - docker push $ECR_REGISTRY/my-app:$IMAGE_TAG
    - docker push $ECR_REGISTRY/my-app:latest
  rules:
    - if: '$CI_COMMIT_BRANCH =~ /^(main|develop)$/'
✅
Set AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, and AWS_DEFAULT_REGION in Settings → CI/CD → Variables. Always mask sensitive values.

OIDC — Recommended for Production

Instead of storing long-lived AWS keys, use OpenID Connect. GitLab exchanges a short-lived JWT for temporary AWS credentials per job — much more secure.

YAML OIDC authentication 
deploy:
  id_tokens:
    GITLAB_OIDC_TOKEN:
      aud: https://gitlab.com
  before_script:
    - pip install awscli -q
    - |
      STS=$(aws sts assume-role-with-web-identity \
        --role-arn $AWS_ROLE_ARN \
        --role-session-name gitlab-ci-$CI_JOB_ID \
        --web-identity-token $GITLAB_OIDC_TOKEN \
        --output json)
      export AWS_ACCESS_KEY_ID=$(echo $STS | python3 -c \
        "import json,sys; print(json.load(sys.stdin)['Credentials']['AccessKeyId'])")
      export AWS_SECRET_ACCESS_KEY=$(echo $STS | python3 -c \
        "import json,sys; print(json.load(sys.stdin)['Credentials']['SecretAccessKey'])")
      export AWS_SESSION_TOKEN=$(echo $STS | python3 -c \
        "import json,sys; print(json.load(sys.stdin)['Credentials']['SessionToken'])")

Deploy to ECS Fargate

New Image in ECR
→
Fetch Task Def
→
Update Image
→
Register New Revision
→
Update Service → Rolling Deploy
BASH scripts/ecs-deploy.sh 
#!/bin/sh
# Usage: ./ecs-deploy.sh <cluster> <service> <task-family> <image>
set -e

CLUSTER=$1; SERVICE=$2; TASK_FAMILY=$3; IMAGE=$4

# 1. Fetch current task definition
TASK_DEF=$(aws ecs describe-task-definition \
  --task-definition $TASK_FAMILY \
  --query 'taskDefinition' --output json)

# 2. Swap the image
NEW_TASK_DEF=$(echo $TASK_DEF | python3 -c "
import json, sys
td = json.load(sys.stdin)
td['containerDefinitions'][0]['image'] = '$IMAGE'
for key in ['taskDefinitionArn','revision','status','requiresAttributes',
            'compatibilities','registeredAt','registeredBy']:
    td.pop(key, None)
print(json.dumps(td))
")

# 3. Register new revision
NEW_ARN=$(aws ecs register-task-definition \
  --cli-input-json "$NEW_TASK_DEF" \
  --query 'taskDefinition.taskDefinitionArn' --output text)

# 4. Update service → triggers rolling deploy
aws ecs update-service \
  --cluster $CLUSTER --service $SERVICE \
  --task-definition $NEW_ARN --force-new-deployment

# 5. Wait for stability
aws ecs wait services-stable --cluster $CLUSTER --services $SERVICE
echo "✅ Deploy complete"

Multi-Environment Promotion

The gold standard for production CI/CD. Code flows through environments in a controlled way — each promotion requires the previous environment to be healthy.

feature branch
→
develop
→
main
1
Build & Test
Every push builds the Docker image and runs unit tests. Security scan runs on ECR image. All must pass before deployment.
2
Deploy to Dev (automatic)
Every push to develop automatically deploys to the dev environment. Fast feedback loop for developers.
3
Smoke Tests on Dev
Automated smoke tests run against the dev environment. Newman (Postman CLI) hits key endpoints. Staging is blocked until these pass.
4
Deploy to Staging (automatic if dev tests pass)
Only deploys if smoke tests on dev passed. Uses needs to create the dependency gate between stages.
5
Integration + Performance Tests on Staging
Full integration test suite + k6 performance baseline. Production is blocked until both pass.
6
Deploy to Production (manual ✋)
when: manual — a human must click Play in the GitLab UI. With environment protection, multiple approvers can be required.
YAML Promotion pipeline stages 
stages:
  - build
  - test
  - deploy-dev
  - validate-dev
  - deploy-staging
  - validate-staging
  - deploy-production

deploy-dev:
  stage: deploy-dev
  needs: [build-image, unit-tests]
  environment:
    name: dev
    url: https://dev.myapp.com
  rules:
    - if: '$CI_COMMIT_BRANCH == "develop"'

smoke-test-dev:
  stage: validate-dev
  needs: [deploy-dev]
  script:
    - newman run tests/smoke.json --env-var "base_url=https://dev.myapp.com"

deploy-staging:
  stage: deploy-staging
  needs:
    - job: smoke-test-dev   # blocked until smoke tests pass
      artifacts: false

deploy-production:
  stage: deploy-production
  needs:
    - job: smoke-test-staging
      artifacts: false
    - job: performance-test
      artifacts: false
  when: manual                # ✋ human approval
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'

Terraform Explained

Terraform is a universal remote control for cloud infrastructure. You write declarative code describing what you want — Terraform figures out how to create, update, or destroy resources to match.

The 3-Step Cycle

terraform init
→
terraform plan
→
terraform apply
🧱
Resources
The things you create. resource "aws_vpc" "main" { ... } — every resource has a type and local name.
📥
Variables
Inputs to your config. Referenced with var.name. Can have types, defaults, and validation rules.
📤
Outputs
Values you export after apply. Used to display results and share values between modules.
🔍
Data Sources
Read existing resources you didn't create. data "aws_vpc" "existing" { ... } — referenced with data.
🗃️
State
Terraform's memory. Stores what currently exists. Store remotely in S3 + DynamoDB for team use and locking.
🧩
Modules
Reusable folders of .tf files. Call like a function with inputs and outputs. Keeps large infra organized.

Terraform VPC & Networking

HCL modules/vpc/main.tf 
data "aws_availability_zones" "available" {
  state = "available"
}

locals {
  azs             = slice(data.aws_availability_zones.available.names, 0, 2)
  public_subnets  = [for i, _ in local.azs : cidrsubnet(var.vpc_cidr, 8, i)]
  private_subnets = [for i, _ in local.azs : cidrsubnet(var.vpc_cidr, 8, i + 10)]
}

resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_hostnames = true
  enable_dns_support   = true
}

resource "aws_subnet" "public" {
  count                   = length(local.azs)
  vpc_id                  = aws_vpc.main.id
  cidr_block              = local.public_subnets[count.index]
  availability_zone       = local.azs[count.index]
  map_public_ip_on_launch = true   # what makes it "public"
}

resource "aws_subnet" "private" {
  count             = length(local.azs)
  vpc_id            = aws_vpc.main.id
  cidr_block        = local.private_subnets[count.index]
  availability_zone = local.azs[count.index]
}

# NAT Gateway — private subnets reach internet, not the other way
resource "aws_nat_gateway" "main" {
  count         = length(local.azs)
  allocation_id = aws_eip.nat[count.index].id
  subnet_id     = aws_subnet.public[count.index].id   # lives in public subnet
  depends_on    = [aws_internet_gateway.main]
}

Terraform ECS Fargate

HCL modules/ecs/main.tf 
resource "aws_ecs_service" "main" {
  name            = "${var.project_name}-${var.environment}"
  cluster         = aws_ecs_cluster.main.id
  task_definition = aws_ecs_task_definition.main.arn
  desired_count   = var.desired_count
  launch_type     = "FARGATE"

  network_configuration {
    subnets          = aws_subnet.private[*].id
    security_groups  = [aws_security_group.ecs_tasks.id]
    assign_public_ip = false
  }

  load_balancer {
    target_group_arn = aws_lb_target_group.main.arn
    container_name   = var.project_name
    container_port   = var.container_port
  }

  # Rolling deploy: keep 50% healthy, allow 200% during deploy
  deployment_minimum_healthy_percent = 50
  deployment_maximum_percent         = 200

  # Auto rollback if new tasks fail health checks
  deployment_circuit_breaker {
    enable   = true
    rollback = true
  }

  # Let CI/CD manage image updates — Terraform manages infra only
  lifecycle {
    ignore_changes = [task_definition]
  }

  depends_on = [aws_lb_listener.http]
}

# Auto-scale on CPU — scale out fast, scale in slowly
resource "aws_appautoscaling_policy" "cpu" {
  target_tracking_scaling_policy_configuration {
    predefined_metric_specification {
      predefined_metric_type = "ECSServiceAverageCPUUtilization"
    }
    target_value       = 70.0
    scale_in_cooldown  = 300
    scale_out_cooldown = 60
  }
}

Terraform in CI/CD

YAML .gitlab-ci.yml (Terraform pipeline) 
stages:
  - validate
  - plan
  - apply

default:
  image: hashicorp/terraform:1.7.0
  before_script:
    - cd terraform
    - terraform init

validate:
  stage: validate
  script:
    - terraform validate
    - terraform fmt -check -recursive
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'

plan:
  stage: plan
  script:
    - terraform plan -var="environment=$TF_ENV" -out=plan.tfplan
    - terraform show -no-color plan.tfplan > plan.txt
  artifacts:
    paths: [terraform/plan.tfplan, terraform/plan.txt]
    expire_in: 1 week

apply-staging:
  stage: apply
  needs: [plan]
  script:
    - terraform apply -auto-approve -var="environment=staging"
  rules:
    - if: '$CI_COMMIT_BRANCH == "develop"'

apply-production:
  stage: apply
  needs: [plan]
  script:
    - terraform apply -auto-approve -var="environment=production"
  when: manual    # ✋ human approves infra changes
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'

PYTHON weather.py (key pattern) 
import os, requests

API_KEY = os.getenv("OPENWEATHER_API_KEY")

def get_weather(city: str) -> dict:
    if not API_KEY:
        raise EnvironmentError("OPENWEATHER_API_KEY not set")

    response = requests.get(
        "https://api.openweathermap.org/data/2.5/weather",
        params={"q": city, "appid": API_KEY, "units": "metric"},
        timeout=10
    )
    response.raise_for_status()   # raises on 4xx/5xx
    return response.json()

try:
    data = get_weather("London")
    print(f"Temp: {data['main']['temp']}°C")
except requests.exceptions.HTTPError as e:
    print(f"City not found: {e}")

PYTHON FastAPI pattern 
from fastapi import FastAPI, HTTPException, Depends
from sqlalchemy.ext.asyncio import AsyncSession
from pydantic import BaseModel, Field

app = FastAPI(title="Task API")

class TaskCreate(BaseModel):
    title: str = Field(..., min_length=1, max_length=100)
    priority: str = "medium"

# Dependency injection — db session per request
async def get_db():
    async with AsyncSessionLocal() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise

@app.post("/tasks/", status_code=201)
async def create_task(task: TaskCreate, db: AsyncSession = Depends(get_db)):
    db_task = Task(**task.model_dump())
    db.add(db_task)
    await db.flush()
    await db.refresh(db_task)
    return db_task

PYTHON Celery worker pattern 
from celery import Celery

app = Celery(
    "tasks",
    broker="redis://localhost:6379/0",
    backend="redis://localhost:6379/1"
)

# bind=True gives access to self for retry
@app.task(bind=True, max_retries=3, default_retry_delay=60)
def send_email(self, to: str, subject: str, body: str):
    try:
        # ... send email logic ...
        return {"status": "sent", "to": to}
    except Exception as exc:
        raise self.retry(exc=exc)   # auto-retry with delay

# Dispatch async — non-blocking
send_email.apply_async(
    args=["user@example.com", "Hello", "Body"],
    countdown=5   # run in 5 seconds
)
