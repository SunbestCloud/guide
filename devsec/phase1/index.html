P-01

SAST Integration — Static Code Analysis in CI

Integrate Semgrep and SonarQube into GitLab CI to scan every commit for security vulnerabilities before they reach your main branch.

Semgrep SonarQube GitLab CI Python/Node App
● BEGINNER FRIENDLY
⏱ Estimated: 3–5 hours
▼
ℹ️

What is SAST? Static Application Security Testing analyzes source code without running it. It catches vulnerabilities like SQL injection, hardcoded credentials, insecure deserialization, and dangerous function calls at the code level, before a single line executes in production.

1

Set Up a Vulnerable Sample Application

Use a deliberately vulnerable Python Flask app to see SAST in action. This gives you real findings to work with.

bash
# Clone a vulnerable test app
git clone https://github.com/OWASP/WebGoat.git
# Or create a simple Python Flask app with intentional vulnerabilities
mkdir devsecops-lab && cd devsecops-lab
git init
python3 -m venv venv && source venv/bin/activate
pip install flask sqlalchemy
python — app.py (intentionally vulnerable)
from flask import Flask, request
import sqlite3, subprocess, os

app = Flask(__name__)

# ❌ VULNERABILITY 1: SQL Injection
@app.route('/user')
def get_user():
    username = request.args.get('name')
    conn = sqlite3.connect('users.db')
    # Never do this — direct string concatenation
    query = f"SELECT * FROM users WHERE name = '{username}'"
    return conn.execute(query).fetchall()

# ❌ VULNERABILITY 2: OS Command Injection
@app.route('/ping')
def ping():
    host = request.args.get('host')
    result = subprocess.run(f"ping -c 1 {host}", shell=True)
    return str(result)

# ❌ VULNERABILITY 3: Hardcoded secret
SECRET_KEY = "super-secret-password-123"
AWS_KEY    = "AKIAIOSFODNN7EXAMPLE"

Create GitLab CI SAST Stage

yaml — .gitlab-ci.yml
stages:
  - security
  - test
  - build
  - deploy

# ── SAST with Semgrep ─────────────────────────────────
semgrep-sast:
  stage: security
  image: returntocorp/semgrep:latest
  script:
    - semgrep --config=auto
              --config=p/owasp-top-ten
              --config=p/python
              --json
              --output semgrep-report.json
              ./src
    # Fail pipeline if HIGH or CRITICAL severity found
    - |
      python3 <<EOF
      import json, sys
      with open("semgrep-report.json") as f:
          data = json.load(f)
      critical = [r for r in data["results"]
                  if r["extra"]["severity"] in ["ERROR", "WARNING"]]
      if critical:
          print(f"BLOCKING: {len(critical)} high-severity findings")
          sys.exit(1)
      print("SAST passed: no high-severity issues")
      EOF
  artifacts:
    when: always
    reports:
      sast: semgrep-report.json
    paths:
      - semgrep-report.json
    expire_in: 7 days
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
    - if: '$CI_COMMIT_BRANCH == "main"'

# ── SAST with Bandit (Python specific) ───────────────
bandit-sast:
  stage: security
  image: python:3.11-slim
  script:
    - pip install bandit
    - bandit -r ./src -f json -o bandit-report.json -ll
    - |
      python3 <<EOF
      import json, sys
      with open("bandit-report.json") as f:
          data = json.load(f)
      high = [i for i in data["results"]
              if i["issue_severity"] == "HIGH"]
      print(f"Bandit findings - HIGH: {len(high)}")
      if len(high) > 0:
          for i in high:
              print(f"  {i['issue_text']} at {i['filename']}:{i['line_number']}")
          sys.exit(1)
      EOF
  artifacts:
    when: always
    paths: [bandit-report.json]
    expire_in: 7 days

Add SonarQube Quality Gate to Pipeline

yaml — append to .gitlab-ci.yml
sonarqube-scan:
  stage: security
  image: sonarsource/sonar-scanner-cli:latest
  variables:
    SONAR_USER_HOME: "${CI_PROJECT_DIR}/.sonar"
    GIT_DEPTH: "0"
  cache:
    key: "${CI_JOB_NAME}"
    paths: [.sonar/cache]

    - sonar-scanner
        -Dsonar.projectKey=$CI_PROJECT_NAME
        -Dsonar.sources=.
        -Dsonar.host.url=$SONAR_HOST_URL
        -Dsonar.token=$SONAR_TOKEN
        -Dsonar.qualitygate.wait=true
  allow_failure: false

P-02

Secrets Detection — Pre-commit Hooks + Pipeline Scanning

Stop API keys, AWS credentials, private keys, and tokens from ever reaching your repository. Pre-commit hooks + TruffleHog + GitLeaks in CI.

TruffleHog GitLeaks pre-commit detect-secrets
● BEGINNER FRIENDLY
⏱ Estimated: 2–4 hours
▼
⚠️

Why this matters: In 2023, over 10 million secrets were exposed in public GitHub repos. One leaked AWS key can cost thousands in unauthorized charges within minutes. Secrets scanning is the highest ROI security control you can add — it takes 2 hours to set up and prevents catastrophic breaches.

1

Install Pre-commit Framework

Pre-commit hooks run before any commit is recorded. The secret never enters git history.

bash
pip install pre-commit detect-secrets

# Initialize in your repo
cd your-repo
pre-commit install
yaml — .pre-commit-config.yaml
repos:
  # detect-secrets baseline approach
  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.4.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']

  # GitLeaks pre-commit hook
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.18.0
    hooks:
      - id: gitleaks

  # General hygiene
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.5.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-added-large-files
        args: ['--maxkb=500']
bash — Initialize baseline
# Create initial secrets baseline (marks known false positives)
detect-secrets scan > .secrets.baseline
git add .secrets.baseline .pre-commit-config.yaml
git commit -m "chore: add secrets detection pre-commit hooks"

# Test it — try committing a fake key
echo "AWS_SECRET=AKIAIOSFODNN7EXAMPLE" > test.py
git add test.py
git commit -m "test"
# → Should be BLOCKED by pre-commit
2

GitLeaks Configuration File

Define custom regex rules for your org-specific secrets (internal API keys, DB connection strings, etc.)

toml — .gitleaks.toml
[extend]
useDefault = true

[[rules]]
id = "custom-internal-api-key"
description = "Internal API Key"
regex = '''MYORG-[A-Z0-9]{32}'''
tags = ["key", "internal"]

[[rules]]
id = "db-connection-string"
description = "Database connection string with credentials"
regex = '''(postgres|mysql|mongodb)\:\/\/[^:]+:[^@]+@'''
tags = ["database", "credentials"]

[allowlist]
description = "Global Allowlist"
regexes = [
  '''EXAMPLE_KEY''',          # Documentation examples
  '''test_secret_value'''     # Unit test fixtures
]
paths = [
  '''tests/fixtures/''',
  '''docs/examples/'''
]
3

TruffleHog in GitLab CI

TruffleHog uses entropy analysis + 800+ regex patterns. It can scan your entire git history, not just the latest commit.

yaml — .gitlab-ci.yml
trufflehog-scan:
  stage: security
  image: trufflesecurity/trufflehog:latest
  script:
    # Scan entire git history for verified secrets
    - trufflehog git file://. --only-verified --json > trufflehog-report.json || true
    - |
      python3 <<EOF
      import json, sys
      findings = []
      with open("trufflehog-report.json") as f:
          for line in f:
              if line.strip():
                  findings.append(json.loads(line))
      if findings:
          print(f"CRITICAL: {len(findings)} verified secrets found!")
          for s in findings:
              print(f"  Type: {s.get('DetectorName', 'unknown')}")
              print(f"  File: {s.get('SourceMetadata', {}).get('Data', {})}")
          sys.exit(1)
      print("No verified secrets found.")
      EOF
  artifacts:
    when: always
    paths: [trufflehog-report.json]
    expire_in: 7 days

gitleaks-scan:
  stage: security
  image: zricethezav/gitleaks:latest
  script:
    - gitleaks detect
        --source .
        --config .gitleaks.toml
        --report-format json
        --report-path gitleaks-report.json
        --exit-code 1
  artifacts:
    when: always
    paths: [gitleaks-report.json]

Scan Historical Git History

If you're adding this to an existing repo, you need to check what's already committed.

bash
# Scan all git history
docker run --rm -v "$(pwd):/repo" \
  trufflesecurity/trufflehog:latest \
  git file:///repo --only-verified

# If secrets found in history, you MUST rotate them AND rewrite history
# Use git-filter-repo to remove secrets from history
pip install git-filter-repo

git filter-repo \
  --path-regex '.*\.env' \
  --invert-paths \
  --force

# Then force push ALL branches (coordinate with team)
git push origin --all --force
🚨

If a secret is found in git history, assume it is compromised. Rotate the credential immediately — even if the repo was private. History rewrites do not help if anyone cloned the repo while the secret was present.

P-03

SCA — Software Composition Analysis

Automatically find CVEs in your third-party dependencies (npm, pip, Maven, etc.) before they reach production. OWASP Dependency-Check + Snyk.

OWASP Dep-Check Snyk Trivy FS CVSS Scoring
● BEGINNER FRIENDLY
⏱ Estimated: 3–4 hours
▼
ℹ️

Log4Shell context: The Log4j vulnerability (CVE-2021-44228) affected millions of applications. SCA tools detect exactly this — a vulnerable version of a transitive dependency buried 4 layers deep in your dependency tree. Without SCA, you wouldn't know until your app is being exploited.

1

OWASP Dependency-Check Setup

bash — Local run
# Run against your project directory
docker run --rm \
  -v $(pwd):/src \
  -v $(pwd)/reports:/report \
  owasp/dependency-check \
  --project "MyApp" \
  --scan /src \
  --format HTML \
  --format JSON \
  --out /report \
  --failOnCVSS 7

# Output: /report/dependency-check-report.html
# Open in browser to see all CVEs with CVSS scores
2

SCA in GitLab CI Pipeline

yaml — .gitlab-ci.yml
dependency-check:
  stage: security
  image: owasp/dependency-check:latest
  script:
    - /usr/share/dependency-check/bin/dependency-check.sh
        --project "$CI_PROJECT_NAME"
        --scan .
        --format JSON
        --format HTML
        --out ./reports
        --failOnCVSS 7.0
        --enableRetired
  artifacts:
    when: always
    paths: [reports/]
    expose_as: "OWASP Dependency Check Report"
    expire_in: 30 days

# Trivy filesystem scan (faster, great SBOM support)
trivy-fs-scan:
  stage: security
  image: aquasec/trivy:latest
  script:
    - trivy fs
        --exit-code 0
        --severity HIGH,CRITICAL
        --format json
        --output trivy-fs-report.json
        .
    - |
      python3 <<EOF
      import json, sys
      with open("trivy-fs-report.json") as f:
          data = json.load(f)
      critical = []
      for result in data.get("Results", []):
          for vuln in result.get("Vulnerabilities", []) or []:
              if vuln["Severity"] == "CRITICAL":
                  critical.append(vuln)
      if critical:
          print(f"BLOCKING: {len(critical)} CRITICAL CVEs in dependencies")
          for v in critical[:5]:
              print(f"  {v['VulnerabilityID']}: {v['PkgName']}@{v.get('InstalledVersion', '?')}")
          sys.exit(1)
      EOF
  artifacts:
    when: always
    paths: [trivy-fs-report.json]

P-04

Container Image Scanning — Trivy + Clair

Every Docker image you build gets scanned for OS-level CVEs, misconfigurations, and vulnerable packages before it's pushed to your registry or deployed.

Trivy Clair Docker Registry Security
◆ INTERMEDIATE
⏱ Estimated: 4–6 hours
▼
1

Build a Deliberately Vulnerable Image

dockerfile — Dockerfile.vulnerable
# ❌ Old base image with known CVEs
FROM ubuntu:18.04

# ❌ Running as root
USER root

# ❌ Installing vulnerable packages
RUN apt-get update && apt-get install -y \
    openssl=1.1.1-1ubuntu2.1~18.04.20 \
    curl \
    python3 \
    python3-pip

# ❌ Hardcoded credentials in image layer
ENV DB_PASSWORD=mysecretpassword

COPY . /app
RUN pip3 install -r /app/requirements.txt

CMD ["python3", "/app/app.py"]
dockerfile — Dockerfile.secure (fixed)
# ✅ Use minimal, current base image
FROM python:3.11-alpine AS builder

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

FROM python:3.11-alpine AS runtime

# ✅ Non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

WORKDIR /app
COPY --from=builder /usr/local/lib/python3.11/site-packages /usr/local/lib/python3.11/site-packages
COPY --chown=appuser:appgroup . .

# ✅ No secrets in image — use runtime env injection
# DB_PASSWORD passed at runtime via AWS Secrets Manager or env

EXPOSE 8080
CMD ["python3", "app.py"]
2

Trivy Container Scan — Local

bash
# Install Trivy
docker pull aquasec/trivy:latest

# Build and scan the vulnerable image
docker build -f Dockerfile.vulnerable -t myapp:vulnerable .
docker run --rm -v /var/run/docker.sock:/var/run/docker.sock \
  aquasec/trivy image \
  --severity HIGH,CRITICAL \
  --format table \
  myapp:vulnerable

# Now scan the secure version
docker build -f Dockerfile.secure -t myapp:secure .
docker run --rm -v /var/run/docker.sock:/var/run/docker.sock \
  aquasec/trivy image \
  --severity HIGH,CRITICAL \
  myapp:secure

# Compare the CVE counts — dramatic difference

# Also scan for misconfigurations in Dockerfile
docker run --rm -v "$(pwd):/src" aquasec/trivy config /src

Container Scanning in GitLab CI

yaml — .gitlab-ci.yml
build-image:
  stage: build
  image: docker:24
  services: [docker:24-dind]
  script:
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA .
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA

container-scan:
  stage: build
  image: aquasec/trivy:latest
  needs: [build-image]
  variables:
    TRIVY_EXIT_CODE: "1"
    TRIVY_SEVERITY: "HIGH,CRITICAL"
    TRIVY_NO_PROGRESS: "true"
  script:
    # Cache Trivy DB between runs
    - mkdir -p /tmp/trivy-cache
    - trivy image
        --cache-dir /tmp/trivy-cache
        --exit-code $TRIVY_EXIT_CODE
        --severity $TRIVY_SEVERITY
        --ignore-unfixed
        --format json
        --output trivy-image-report.json
        $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA
  cache:
    key: trivy-db
    paths: [/tmp/trivy-cache/]
  artifacts:
    when: always
    reports:
      container_scanning: trivy-image-report.json
    paths: [trivy-image-report.json]

# Only push to registry AFTER scan passes
promote-image:
  stage: deploy
  needs: [container-scan]
  script:
    - docker tag $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA
               $CI_REGISTRY_IMAGE:latest
    - docker push $CI_REGISTRY_IMAGE:latest
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
P-05

DAST — Dynamic Application Security Testing with OWASP ZAP

Attack your own staging environment automatically. Find vulnerabilities that only appear at runtime: XSS, SQLi, auth bypass, CSRF, insecure headers.

OWASP ZAP Nikto Active Scanning API Security
◆ INTERMEDIATE
⏱ Estimated: 5–7 hours
▼
ℹ️

SAST vs DAST: SAST reads your code. DAST runs your application and throws real attack payloads at it — SQL injection strings, XSS scripts, path traversals. It finds vulnerabilities that only appear at runtime, like misconfigured frameworks or insecure third-party integrations.

1

bash
# Baseline: passive scan only (no active attacks) — safe for production
docker run --rm \

DAST in GitLab CI Pipeline

yaml — .gitlab-ci.yml
deploy-staging:
  stage: deploy
  script:
    - # Your deployment commands here
    - echo $STAGING_URL > staging_url.txt
  artifacts:

dast-baseline:

  needs: [deploy-staging]

          for a in high_alerts:
              print(f"  {a['alert']}: {a['desc'][:100]}")
          sys.exit(1)
      print("DAST baseline passed!")
      EOF

Policy as Code — OPA / Open Policy Agent

Define security decisions as code using Rego language. Enforce that images come from approved registries, all containers run non-root, and no CRITICAL CVEs pass through.

OPA Rego Conftest Policy Gates
▲ ADVANCED
⏱ Estimated: 6–8 hours
▼
ℹ️

What is OPA? Open Policy Agent is a general-purpose policy engine. You write policies in Rego (a declarative language). OPA decides: should this Docker image be deployed? Should this Terraform plan be applied? Should this API request be allowed? It's the enforcement brain for all your security rules.

1

Install Conftest (OPA for CI/CD)

bash
brew install conftest   # macOS
# or
wget https://github.com/open-policy-agent/conftest/releases/download/v0.46.0/conftest_0.46.0_Linux_x86_64.tar.gz
tar xzf conftest*.tar.gz && mv conftest /usr/local/bin/
2

Write Security Policies in Rego

rego — policy/docker.rego
package main

# ── Policy 1: No running as root ─────────────────────
deny[msg] {
  input.config.User == ""
  msg := "Dockerfile must specify a non-root USER"
}

deny[msg] {
  input.config.User == "root"
  msg := "Dockerfile must not run as root user"
}

# ── Policy 2: Base image must be approved ────────────
approved_base_images := {
  "python:3.11-alpine",
  "node:20-alpine",
  "golang:1.21-alpine",
  "ubuntu:22.04"
}

deny[msg] {
  image := input.config.Image
  not approved_base_images[image]
  msg := sprintf("Base image '%v' is not in approved list", [image])
}

# ── Policy 3: No hardcoded secrets in ENV ────────────
secret_patterns := [
  "password", "secret", "key", "token", "credential"
]

deny[msg] {
  env_var := input.config.Env[_]
  parts := split(env_var, "=")
  key := lower(parts[0])
  pattern := secret_patterns[_]
  contains(key, pattern)
  count(parts) > 1
  parts[1] != ""
  msg := sprintf("ENV var '%v' may contain a hardcoded secret", [parts[0]])
}

# ── Policy 4: HEALTHCHECK must be defined ────────────
deny[msg] {
  not input.config.Healthcheck
  msg := "Dockerfile must define a HEALTHCHECK"
}
rego — policy/trivy.rego (enforce CVE threshold)
package main

deny[msg] {
  result := input.Results[_]
  vuln := result.Vulnerabilities[_]
  vuln.Severity == "CRITICAL"
  msg := sprintf(
    "CRITICAL CVE %v in %v@%v — must be fixed before deploy",
    [vuln.VulnerabilityID, vuln.PkgName, vuln.InstalledVersion]
  )
}

warn[msg] {
  result := input.Results[_]
  vuln := result.Vulnerabilities[_]
  vuln.Severity == "HIGH"
  msg := sprintf(
    "HIGH CVE %v in %v — consider updating",
    [vuln.VulnerabilityID, vuln.PkgName]
  )
}

P-08

SBOM — Software Bill of Materials with Syft + Grype

Generate a complete inventory of every component in your software. SPDX and CycloneDX format SBOMs fed into Grype for continuous CVE matching across your entire fleet.

Syft Grype SPDX CycloneDX
▲ ADVANCED
⏱ Estimated: 4–6 hours
▼
ℹ️

Why SBOM? US Executive Order 14028 (2021) mandates SBOMs for software sold to the federal government. Even outside government, an SBOM lets you answer in minutes: "Are any of our 200 services using the vulnerable version of OpenSSL?" Without it, that answer takes weeks.

1

Generate SBOM with Syft

bash
# Install
curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh

# Generate SBOM from a container image
syft myapp:latest -o spdx-json > sbom.spdx.json
syft myapp:latest -o cyclonedx-json > sbom.cyclonedx.json
syft myapp:latest -o table   # human-readable

# Generate from directory (source code)
syft dir:./src -o spdx-json > sbom-source.spdx.json

# Count components
cat sbom.spdx.json | python3 -c "
import json, sys
d = json.load(sys.stdin)
pkgs = d.get('packages', [])
print(f'Total components: {len(pkgs)}')
by_type = {}
for p in pkgs:
    t = p.get('externalRefs', [{}])[0].get('referenceCategory', 'OTHER')
    by_type[t] = by_type.get(t, 0) + 1
for k, v in sorted(by_type.items(), key=lambda x: -x[1]):
    print(f'  {k}: {v}')
"
2

Scan SBOM with Grype

bash
# Install Grype
curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh | sh

# Scan SBOM for CVEs
grype sbom:./sbom.spdx.json
grype sbom:./sbom.spdx.json --fail-on critical

# JSON output for automation
grype sbom:./sbom.spdx.json -o json > grype-results.json

# Parse results
cat grype-results.json | python3 -c "
import json, sys
d = json.load(sys.stdin)
matches = d.get('matches', [])
by_sev = {}
for m in matches:
    s = m['vulnerability']['severity']
    by_sev[s] = by_sev.get(s, 0) + 1
print('CVE Summary:')
for sev in ['Critical', 'High', 'Medium', 'Low']:
    print(f'  {sev}: {by_sev.get(sev, 0)}')
"
3

SBOM in GitLab CI + Attestation

yaml — .gitlab-ci.yml
generate-sbom:
  stage: sbom
  image: anchore/syft:latest
  script:
    # Generate in multiple formats
    - syft $IMAGE -o spdx-json  > sbom.spdx.json
    - syft $IMAGE -o cyclonedx-json > sbom.cyclonedx.json

    - grype sbom:./sbom.spdx.json
            --fail-on critical
            -o json > grype-results.json

    # Attach SBOM to image as OCI artifact
    - cosign attach sbom --sbom sbom.spdx.json $IMAGE
  artifacts:
    paths: [sbom.spdx.json, sbom.cyclonedx.json, grype-results.json]
    expire_in: 90 days

# Sign the SBOM with Cosign (supply chain security)
sign-attestation:
  stage: sbom
  needs: [generate-sbom]
  image: gcr.io/projectsigstore/cosign:latest
  script:
    - cosign sign --key cosign.key $IMAGE
    - cosign attest --predicate sbom.cyclonedx.json
                    --type cyclonedx
                    --key cosign.key
                    $IMAGE
