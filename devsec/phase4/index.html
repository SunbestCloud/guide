SIEM 101 — Every Concept Explained Simply

Read this before touching any tool. These mental models make everything else click.

📋 What Is a Log? (Simpler Than You Think)

A log is just a timestamped record of something that happened. Every system — your Linux server, your AWS account, your application, your firewall — generates a constant stream of these records. Most of the time they're boring. Sometimes one log record is the difference between catching a breach in 4 minutes or 4 months.

Example log line from CloudTrail: 2024-01-15T03:47:22Z — User "arn:aws:iam::123:user/dev-alice" called "DeleteBucket" on "s3:::prod-customer-data" from IP 185.220.101.42. That single line tells you: who (dev-alice), what (deleted S3 bucket), where (prod-customer-data), when (3:47am), and from where (a Tor exit node IP). That's an incident. Logs tell the story.

🏗️ What Is a SIEM?

SIEM = Security Information and Event Management. It's a system that: (1) Collects logs from everywhere (servers, firewalls, AWS, K8s, apps). (2) Stores them in a searchable database. (3) Normalizes them into a consistent format (different systems log differently). (4) Correlates events across sources (this login + that network connection + this file access = suspicious). (5) Alerts when patterns match known attack behaviours.

Think of it as the brain of your Security Operations Centre (SOC). Without a SIEM, every alert tool shouts at you separately. With one, you see the full picture: GuardDuty found a suspicious EC2 instance, CloudTrail shows it called unusual APIs, VPC Flow Logs show it making outbound connections to a known C2 server — all correlated in one view.

⚠️ The Log Sources You MUST Collect (AWS Focus)

CloudTrail: Every AWS API call. Who created what, deleted what, changed permissions. This is the most critical log source in AWS. A CloudTrail-less AWS account is a blind AWS account. Enable in all regions, multi-account.

VPC Flow Logs: Network traffic metadata for every ENI. Source IP, destination IP, port, bytes, accept/reject. You already set these up in Phase 2. Now you'll query them for threat hunting.

GuardDuty Findings: ML-based threat detections — treated as high-quality pre-built alerts. Feed directly into your SIEM.

Linux Auditd: System call level logging on EC2. What files were accessed, what processes ran, what users did what. The deep forensic source.

Application Logs: Auth events, errors, HTTP requests. Failed logins, SQL errors, unexpected endpoints.

K8s Audit Logs: Every kubectl command, API call to your cluster. Who deployed what, who exec'd into which pod.

🎯 What Is a Detection Rule?

A detection rule is a pattern that, when matched in log data, fires an alert. Good detection rules are: specific enough to not fire on normal activity (low false positive rate), broad enough to catch real attacks (high true positive rate), and mapped to the MITRE ATT&CK technique they detect.

Simple example: IF CloudTrail shows "DeleteBucket" AND the calling IP is in the Tor exit node list AND the time is between midnight and 5am → ALERT: P1 - Suspicious S3 Deletion. The art of detection engineering is writing rules like this that actually work in production without flooding the SOC with noise.

🔄 The Incident Response Lifecycle (The 6 Phases)

Every security incident goes through these phases. Your job is to have a plan for each one before an incident happens — because at 3am with systems going down is not the time to figure out what to do.

Phases 1–3: Detection & Analysis

Preparation: Runbooks written, tools deployed, team trained — before anything happens. Detection & Analysis: Alert fires, analyst investigates, confirms it's real (not false positive), determines severity and scope. Containment: Stop the bleeding — isolate affected systems, revoke compromised credentials, block attacker IPs.

Phases 4–6: Recovery & Learning

Eradication: Remove the threat completely — malware cleaned, backdoors closed, vulnerabilities patched. Recovery: Restore systems from clean backups, verify integrity, return to production. Lessons Learned: Post-incident review within 72 hours. What failed? What detection was missed? How do we prevent recurrence?

📊 Key Metrics — How You Measure SOC Effectiveness

MTTD (Mean Time to Detect): Average time from breach start to first alert. Industry average is 197 days. Your goal: under 24 hours, ideally minutes.

MTTR (Mean Time to Respond): From detection to containment. Every minute counts — attacker dwell time is directly correlated to breach cost.

False Positive Rate: Percentage of alerts that aren't real threats. Above 50% and analysts get alert fatigue and start ignoring everything. Target: under 20%.

Alert Coverage: What percentage of MITRE ATT&CK techniques do you have detections for? A mature SOC covers 80%+. Most organisations start at 15%.

The Attacker's Playbook

MITRE ATT&CK Framework

The most important framework in modern defensive security. Understanding this makes you think like an attacker — which makes you a vastly better defender.

🗺️ What Is MITRE ATT&CK?

MITRE ATT&CK is a knowledge base of real attacker behaviours, observed from real-world incidents and threat intelligence. It's organised as a matrix of Tactics (what the attacker is trying to achieve) and Techniques (how they achieve it). Every technique has: a description, real-world examples, detection guidance, and mitigation advice.

Why does this matter? When you write a detection rule, you tag it with the ATT&CK technique it covers. This lets you see: "I have 847 detection rules but they only cover 23% of ATT&CK techniques — I'm blind to lateral movement." It turns security coverage from a feeling into a measurable metric.

🔗 The Attack Chain — Connecting the Dots

Real attacks are chains of techniques, not isolated events. The classic AWS attack chain looks like this: Reconnaissance (attacker scans for exposed S3 buckets or API keys in GitHub) → Initial Access (found a leaked IAM key in a public repo) → Discovery (the key runs ListBuckets, DescribeInstances, GetCallerIdentity) → Privilege Escalation (the key can attach policies, so attacker grants themselves AdministratorAccess) → Persistence (creates a new IAM user with its own key as a backdoor) → Defense Evasion (disables CloudTrail in all regions) → Exfiltration (syncs every S3 bucket to their own account).

The average time for this full chain in AWS: 2 minutes once the attacker has the initial key. Your detection and response must be faster. This is why automated SOAR (Project 10) is not optional — a human cannot respond in 2 minutes at 3am.

📋 The 6-Step Triage Process

Step 1 — Acknowledge: Claim the alert so no two analysts work it simultaneously. Set status to "In Progress."

Step 2 — Context Gather: What triggered it? What asset is involved? Is it business-critical? What time is it? Look at the surrounding log context — 30 minutes before and after.

Step 3 — Validate: Is this a real event or a false positive? Look for corroborating evidence across multiple log sources. One log line proves nothing.

Step 4 — Scope: How big is it? Just one account, one host, one region? Or is there evidence of lateral movement to other systems?

Step 5 — Decide: Escalate to P1 (critical, active threat), P2 (high risk, investigate now), P3 (medium, schedule investigation), Close (false positive, document why).

Step 6 — Document: Write a timeline of what you found, what actions you took, and why. This becomes the incident record and helps improve future detections.

🦊 The Threat Hunting Mindset

A detection rule waits for something to match. A threat hunter asks: "If an attacker is already in my environment, what would their activity look like in my logs?" Then they go looking. This is why threat hunting catches things detection rules miss — attackers know your rules exist and try to evade them, but they can't hide from a smart human looking at raw data with the right questions.

Every hunt follows the same pattern: Hypothesis (what are you looking for?) → Data Source (which logs would show this?) → Query (how do you search for it?) → Baseline (what's normal?) → Anomalies (what deviates from normal?) → Investigate (are anomalies benign or malicious?)
