Nginx Config & Dockerfile

Custom server block with cache headers

config▼
nginx/default.conf
server {
    listen 80;
    root   /usr/share/nginx/html;
    index  index.html;

    # Gzip compression
    gzip on;
    gzip_types text/css application/javascript;

    location / {
        try_files $uri $uri/ =404;
    }

    # Cache static assets for 30 days
    location ~* \.(css|js|png|jpg|ico)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # Custom 404 page
    error_page 404 /404.html;
}

Why Container Security Matters

Common vulnerabilities and what we'll fix

setup▼

Most Docker tutorials run containers as root, embed secrets in environment variables, and use writable filesystems — all security anti-patterns in production.

🚨Common mistakes: Hardcoded passwords in docker-compose.yml, running as root, no image scanning, no resource limits, writable container filesystem.
terminal — scaffold
mkdir secure-app && cd secure-app
touch Dockerfile app.py docker-compose.yml
02

Non-Root User in Dockerfile

Never run as root inside a container

security▼

If a container running as root is compromised, the attacker has root access to the host. Always create a dedicated user.

Dockerfile — hardened
FROM python:3.12-slim

# Create a non-root user
RUN groupadd -r appuser && useradd -r -g appuser appuser

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY --chown=appuser:appuser . .

# Drop to non-root before running
USER appuser

EXPOSE 5000
CMD ["python", "app.py"]
terminal — verify user
docker build -t secure-app .
docker run --rm secure-app whoami
# Should output: appuser (not root!)
03

Docker Secrets

Never pass credentials via environment variables

security▼

Env vars are visible in docker inspect, logs, and to every process in the container. Docker Secrets mount credentials as files in memory only accessible to the intended container.

terminal — create secret
# Initialize Swarm mode (required for secrets)
docker swarm init

# Create a secret from stdin
echo "supersecretpassword" | docker secret create db_password -

# List secrets (values are never shown)
docker secret ls
docker-compose.yml — using secrets
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
    secrets:
      - db_password

secrets:
  db_password:
    external: true   # references the secret we created above
🔐The secret is mounted at /run/secrets/db_password as a file. Postgres reads it via POSTGRES_PASSWORD_FILE. The value is never in env vars or image layers.
04

Read-Only Filesystem & Resource Limits

Lock down the container's runtime attack surface

security▼
docker-compose.yml — hardened runtime
services:
  api:
    image: secure-app
    read_only: true            # container FS is read-only
    tmpfs:
      - /tmp                     # writable temp space in RAM only
    security_opt:
      - no-new-privileges:true  # block privilege escalation
    cap_drop:
      - ALL                      # drop all Linux capabilities
    cap_add:
      - NET_BIND_SERVICE         # re-add only what's needed
    deploy:
      resources:
        limits:
          cpus: '0.5'           # max 50% of one CPU
          memory: 256M
05

Image Scanning with Trivy

Find CVEs in your images before deploying

security▼

Trivy is the industry-standard open-source container scanner — it checks OS packages, language libraries, and config files for known vulnerabilities.

terminal
# Run Trivy itself in Docker — no install needed!
docker run --rm aquasec/trivy image secure-app

# Scan and output only HIGH/CRITICAL CVEs
docker run --rm aquasec/trivy image \
  --severity HIGH,CRITICAL secure-app

# Exit with non-zero if CRITICAL CVEs found (for CI)
docker run --rm aquasec/trivy image \
  --exit-code 1 --severity CRITICAL secure-app
✅Best practice: Run Trivy in your CI pipeline on every build. The --exit-code 1 flag fails the build if critical CVEs are found — enforcing security gates.

  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build-test-push:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Docker Buildx

        uses: docker/build-push-action@v5

          context: .
          target: production
          load: true
          tags: myapp:latest

      - name: Scan with Trivy
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: myapp:latest
          severity: CRITICAL
          exit-code: '1'

      - name: Login to Docker Hub
        if: github.ref == 'refs/heads/main'
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Push to Docker Hub
        if: github.ref == 'refs/heads/main'

          docker tag myapp:latest ${{ secrets.DOCKERHUB_USERNAME }}/myapp:latest

BuildKit Cache & Layer Optimization

Make CI builds 3–5× faster with registry cache

docker▼

By default, CI runners start fresh — no layer cache. BuildKit's registry cache exports cache to Docker Hub and imports it on the next run.

workflow step — with cache
- name: Build with cache
  uses: docker/build-push-action@v5
  with:
    context: .
    target: production
    push: true
    tags: myuser/myapp:latest
    cache-from: type=registry,ref=myuser/myapp:buildcache
    cache-to:   type=registry,ref=myuser/myapp:buildcache,mode=max
⚡With registry cache, a Node.js build that took 3 min drops to ~30s on repeat runs because npm install layers are cached in the registry.
05

Semantic Versioning Tags

Tag images with git SHAs and semver

config▼
workflow — smart tagging
- name: Docker meta
  id: meta
  uses: docker/metadata-action@v5
  with:
    images: myuser/myapp
    tags: |
      type=sha                     # myapp:sha-abc1234
      type=semver,pattern={{version}}  # myapp:1.2.3
      type=raw,value=latest,enable=${{ github.ref == 'refs/heads/main' }}

- name: Build and push
  uses: docker/build-push-action@v5
  with:
    tags:   ${{ steps.meta.outputs.tags }}
    labels: ${{ steps.meta.outputs.labels }}
🏷️Every commit gets a unique SHA tag. Every git tag like v1.2.3 creates a proper semver image. latest only updates on main branch merges.

const express = require('express');
const client  = require('prom-client');

const app = express();
client.collectDefaultMetrics();   // CPU, memory, event loop, etc.

// Custom counter: count HTTP requests by route
const httpReqs = new client.Counter({
  name: 'http_requests_total',
  help: 'Total HTTP requests',
  labelNames: ['method', 'route', 'status']
});

app.use((req, res, next) => {

  next();
});

app.get('/metrics', async(q,r) => {

});

Docker Swarm — Cluster Orchestration

Deploy a multi-service stack across a Swarm cluster; rolling updates, drain nodes, scale services

expert0 / 5
01

Swarm Concepts

Managers, workers, services, tasks, stacks

setup▼

Docker Swarm turns a pool of Docker hosts into a single virtual cluster. A Manager node schedules Services (replicated containers called Tasks) across Worker nodes.

🏗️Key concepts: Service = desired state (3 replicas of myapp). Task = a running container fulfilling that service. Stack = a group of services from a Compose file deployed to Swarm.
💻For this project, a single machine in Swarm mode works fine. For a real cluster, provision 3 VMs (1 manager + 2 workers) on any cloud provider.
02

Initialize Swarm & Add Workers

Create the cluster

docker▼
terminal — manager node
# Initialize Swarm on the manager node
docker swarm init --advertise-addr <MANAGER_IP>
# Outputs a join token — copy it!

# List nodes
docker node ls
terminal — worker nodes
# Run on each worker (token from above)
docker swarm join --token <WORKER_TOKEN> <MANAGER_IP>:2377

# Back on manager — verify all nodes joined
docker node ls
03

Write the Stack File

Compose-style file for Swarm deployment

config▼

Swarm uses docker stack deploy with a Compose file that adds deploy keys for replicas, update policy, placement, and resources.

stack.yml
services:
  web:
    image: nginx:alpine
    ports:
      - "80:80"
    deploy:
      replicas: 3
      update_config:
        parallelism: 1           # update 1 replica at a time
        delay: 10s              # wait 10s between each
        failure_action: rollback # auto-rollback on failure
      restart_policy:
        condition: on-failure
        max_attempts: 3
      placement:
        constraints:
          - node.role == worker   # only run on worker nodes
      resources:
        limits:
          cpus: '0.25'
          memory: 64M

  visualizer:
    image: dockersamples/visualizer:latest
    ports:
      - "8080:8080"
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
    deploy:
      placement:
        constraints: [node.role == manager]  # only on manager
04

Deploy, Scale & Rolling Update

Live operations on the running cluster

run▼
terminal
# Deploy the stack
docker stack deploy -c stack.yml mystack

# List services and their replicas
docker stack services mystack

# Watch tasks running across nodes
docker stack ps mystack

# Scale up web service to 6 replicas
docker service scale mystack_web=6

# Rolling update: swap image
docker service update --image nginx:1.27-alpine mystack_web

# Watch the rolling update live
watch docker service ps mystack_web
🌐Open http://localhost:8080 for the Swarm Visualizer — a real-time visual of which containers are running on which nodes.

Drain a Node & Failover

Simulate a node going down — zero downtime

run▼

Draining removes all tasks from a node and reschedules them elsewhere — used for maintenance without downtime.

terminal
# Drain a worker node (graceful maintenance mode)
docker node update --availability drain <worker-node-id>

# Watch tasks migrate to other nodes in real time
watch docker service ps mystack_web

# Bring the node back to active
docker node update --availability active <worker-node-id>

# Force a rollback if an update goes wrong
docker service rollback mystack_web

# Tear down the stack
docker stack rm mystack
✅While a node is draining, Swarm automatically reschedules its tasks on other available nodes — your service stays up with zero manual intervention.

Docker Network Types

Bridge, host, overlay, macvlan — when to use each

setup▼

Docker networking is one of the most misunderstood areas. Understanding it unlocks true production-grade architectures.

🌐bridge (default): containers on same host can communicate. host: container shares host network stack (no isolation). overlay: multi-host (Swarm). none: fully isolated.
terminal — explore existing networks
docker network ls
docker network inspect bridge   # see default bridge config

# Create a named bridge network
docker network create \
  --driver bridge \
  --subnet 172.20.0.0/16 \
  --gateway 172.20.0.1 \
  mynet

Network Isolation Between Services

Frontend can't reach DB — only backend can

config▼

A real security pattern: put your database on a private network only the backend can reach. The frontend has no route to the database at all.

docker-compose.yml — network isolation
services:
  frontend:
    image: nginx:alpine
    networks: [public]            # only on public network
    ports: ["80:80"]

  backend:
    image: node:20-alpine
    networks: [public, private]   # bridge between both

  database:
    image: postgres:16-alpine
    networks: [private]           # ONLY on private — invisible to frontend
    environment:
      POSTGRES_PASSWORD: secret

networks:
  public:
    driver: bridge
  private:
    driver: bridge
    internal: true               # no external internet access
terminal — verify isolation
docker compose up -d
# Frontend CANNOT reach database:
docker compose exec frontend ping database
# Should fail — database is invisible from public network

# Backend CAN reach both:
docker compose exec backend ping database
# Should succeed

DNS Aliases & Service Discovery

Multiple names for the same container

config▼

DNS aliases let you give a container multiple hostnames — useful for migrating services or running blue-green deployments without changing app code.

docker-compose.yml — DNS aliases
services:
  db-v2:
    image: postgres:16-alpine
    networks:
      private:
        aliases:
          - database         # old name still works
          - db               # short alias
          - postgres.internal  # FQDN style
    environment:
      POSTGRES_PASSWORD: secret

networks:
  private:
🔀This is how zero-downtime database migrations work: keep the old name as an alias while migrating. Apps using the old name continue to work unchanged.

Generate & Mount TLS Certs

Encrypt inter-container traffic with mTLS

security▼

In production, you should encrypt traffic between services — even inside a private Docker network. We'll generate a local CA and self-signed certs.

terminal — generate certs with openssl
mkdir certs && cd certs

# Generate a local CA key and cert
openssl genrsa -out ca.key 4096
openssl req -new -x509 -days 1826 -key ca.key -out ca.crt \
  -subj "/CN=MyLocalCA"

# Generate server key + CSR
openssl genrsa -out server.key 2048
openssl req -new -key server.key -out server.csr \
  -subj "/CN=backend"

# Sign the server cert with our CA
openssl x509 -req -days 365 \
  -in server.csr -CA ca.crt -CAkey ca.key -CAcreateserial \
  -out server.crt
⚠️These are self-signed certs for learning. In production use cert-manager with Let's Encrypt, or a service mesh like Istio / Linkerd for automatic mTLS.

Overlay Networks & Network Policies

Multi-host networking and traffic control

config▼

Overlay networks span multiple Docker hosts in a Swarm. With --opt encrypted, all overlay traffic is automatically encrypted with AES-128-GCM.

terminal — encrypted overlay network
# Create an encrypted overlay network (requires Swarm mode)
docker network create \
  --driver overlay \
  --opt encrypted \
  --attachable \
  secure-overlay

# Deploy service on the encrypted network
docker service create \
  --name secure-api \
  --network secure-overlay \
  --replicas 3 \
  node:20-alpine node -e "require('http').createServer((_,r)=>r.end('ok')).listen(3000)"

# Inspect network — see connected containers and VXLAN config
docker network inspect secure-overlay
terminal — advanced network diagnostics
# Show all IPs assigned to a container
docker inspect --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' <container>

# Packet capture on a container interface (requires tcpdump)
docker run --rm --net container:<id> nicolaka/netshoot tcpdump -i eth0 -n

# Full network connectivity report
docker run --rm --net container:<id> nicolaka/netshoot netstat -tlnp
🎓nicolaka/netshoot is a container packed with every network debugging tool (tcpdump, nmap, curl, dig, netstat, etc.). It's the Swiss Army knife for Docker network troubleshooting.
