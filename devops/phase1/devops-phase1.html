Before Any Code

Linux 101 — How It Actually Works

Read this before opening a terminal. These mental models make everything click faster.

🐧 What Is Linux and Why Does Every Server Run It?

Linux is an operating system kernel — the core software that manages hardware resources (CPU, memory, disk, network) and lets other programs run. When you run Docker, Kubernetes, or any AWS service, you're running on Linux underneath.

Why Linux and not Windows? Three reasons: free and open source (no licensing cost per server), incredibly stable (Linux servers run for years without rebooting), and perfect for automation (everything is a file or a command, making scripting natural). The top 500 supercomputers in the world all run Linux. AWS, GCP, and Azure all run Linux.

📁 The Linux Filesystem — Everything Is a File

Linux has one unified directory tree starting at / (called root). Unlike Windows with C:\, D:\ etc., everything — disks, USB drives, even hardware devices — appears somewhere under /. This is powerful: you interact with hardware the same way you interact with files.

🏢
Analogy: Think of the Linux filesystem like a company building. / is the lobby (everything starts here). /home is the office floor (where users work). /etc is the filing cabinet (configuration files). /var/log is the company's paper trail (logs). /bin and /usr/bin are the tool cupboard (programs you can run). /tmp is the whiteboard (temporary, wiped on reboot).

👤 Users, Groups & Permissions — Linux Security Model

Every file in Linux has three attributes: who owns it (user), what group it belongs to, and what permissions each has. Permissions are: r (read), w (write), x (execute). They're shown as three groups of three: owner / group / everyone.

Example: -rwxr-xr-- means: owner can read+write+execute, group can read+execute, everyone else can only read. The number version: 755 (owner=7=rwx, group=5=r-x, other=5=r-x). You'll set permissions constantly.

The golden rule: never run things as root unless you absolutely have to. Root can delete everything with one command. Use sudo for specific commands that need elevated privileges.

⚙️ Processes — Programs That Are Running Right Now

Every running program is a process with a unique PID (Process ID). Processes have a parent-child hierarchy — your terminal spawned a shell process, which spawns child processes when you run commands. Key commands: ps aux (list all processes), top / htop (live view), kill -9 PID (force stop), systemctl status nginx (check a service).

Systemd is the init system — it's PID 1, the first process, the parent of everything. It manages services (nginx, sshd, your app), starts them on boot, restarts them on crash. You'll interact with it constantly: systemctl start/stop/restart/enable/status servicename.

Shell Scripting

Bash 101 — Automate Everything

Bash is the language of the Linux command line. A Bash script is just a text file full of commands that run in sequence. Master this and repetitive manual work disappears.

🐚 What Bash Actually Is

When you open a terminal, you're talking to a shell — a program that reads your commands and runs them. Bash (Bourne Again Shell) is the most common shell on Linux. A Bash script is just a file where you've written a sequence of commands so you don't have to type them one by one. If you've ever done the same 5 commands more than twice, it should be a script.

📋
Analogy: A Bash script is like a recipe. The recipe (script) tells you exactly what to do step-by-step. You can follow the recipe once, or a hundred times. You can give it to someone else. You can modify one line and get a slightly different result. It's repeatable, shareable, and documented — all the things manual commands are not.

🧩 The Building Blocks of Every Bash Script

Variables store values: NAME="alice", use with $NAME. Conditionals make decisions: if [ condition ]; then ... fi. Loops repeat actions: for file in *.log; do ... done. Functions bundle reusable logic. Exit codes: every command returns 0 (success) or non-zero (failure) — your scripts must check these.

One critical habit: always start scripts with #!/bin/bash (the shebang line — tells the OS which interpreter to use) and set -euo pipefail (stop on first error, treat unset variables as errors, catch pipe failures). This one line prevents 80% of script bugs.

🔧 The Tools Bash Scripts Use Most

grep — search text. grep "ERROR" /var/log/app.log finds all error lines. awk — process columnar data. awk '{print $1, $4}' access.log prints columns 1 and 4. sed — find and replace in files. sed -i 's/old/new/g' file.conf. curl — make HTTP requests from scripts. curl -s https://api.example.com/health. jq — parse JSON. curl ... | jq '.status'. These five tools combined can do 90% of all DevOps scripting tasks.

Networking Foundations

Networking 101 — How Computers Talk

Every DevOps problem eventually becomes a networking problem. Understanding this properly saves hours of debugging.

🌐 The Internet in Three Minutes

Every device on a network has an IP address — a unique identifier like a postal address. Data travels as packets — small chunks, each labelled with source and destination IP. Routers are the post offices — they read the destination address and forward the packet closer to its destination. Ports are like apartment numbers — the IP gets you to the building (server), the port gets you to the right service. Port 80 = HTTP, 443 = HTTPS, 22 = SSH, 5432 = PostgreSQL.

📬
Analogy: An IP address is like a street address. A port is like an apartment number. A packet is like an envelope with "From:" and "To:" fields. TCP is like certified mail — the recipient must sign for every packet. UDP is like regular mail — faster but no delivery confirmation. HTTP is like a specific language two apartments have agreed to use to communicate.

🔍 DNS — The Internet's Phone Book

Computers communicate using IP addresses (like 142.250.80.46), but humans use names (like google.com). DNS (Domain Name System) translates names to IPs. When you type google.com, your computer asks a DNS server "what's the IP for google.com?" and gets back an IP address to connect to.

Key DNS record types: A record — maps hostname to IPv4 address. CNAME — alias from one name to another. MX — mail server for a domain. TXT — arbitrary text, used for verification and SPF. You'll configure all of these when you manage domains in AWS Route 53.

🔒 TCP vs UDP — Reliable vs Fast

TCP (Transmission Control Protocol) is connection-oriented. Before sending data it does a 3-way handshake (SYN → SYN-ACK → ACK). Every packet is acknowledged. Lost packets are retransmitted. Order is guaranteed. This reliability has a cost: latency. Used by HTTP, SSH, PostgreSQL — anything where you need every byte to arrive correctly.

UDP is fire-and-forget. No handshake, no acknowledgment, no guaranteed order. Faster and lower overhead. Used by DNS queries (fast lookups), video streaming (a dropped frame is better than a stutter), and game networking. As a DevOps engineer you'll mostly deal with TCP, but you need to know why something uses each.

Version Control

Git 101 — Track Everything, Break Nothing

Git is not just for developers. Every config file, script, and infrastructure definition you write should be in Git. It's your undo button, your collaboration system, and your audit log.

🌿 What Git Actually Does

Git is a version control system — it tracks every change you make to files over time. You can go back to any previous state, see exactly what changed and when, and work on multiple things simultaneously without breaking each other.

📷
Analogy: Git is like Google Docs' version history, but for code and config files. Every time you "commit," Git takes a snapshot of your files. You can jump back to any snapshot. You can create a "branch" — a parallel copy where you experiment — and merge it back when you're happy. If you break something, you simply revert to the last working snapshot.

🔄 The Git Workflow — 4 States Every File Lives In

Untracked: Git doesn't know about this file yet. Staged: You've told Git "include this change in the next snapshot" with git add. Committed: The snapshot is saved permanently in Git's history with git commit. Pushed: The snapshot is uploaded to a remote server (GitHub, GitLab) with git push.

The typical loop: make changes → git add . → git commit -m "what I did" → git push. Do this after every logical unit of work — not just once a day.

🌳 Branches — Work Without Breaking Things

A branch is a parallel copy of your codebase. The convention: main (or master) is always the working, deployable version. You never push directly to main. Instead: create a branch (git checkout -b feature/add-monitoring), make your changes, push the branch, open a Pull Request (code review), and merge to main when approved. This is how every professional team works.

Key branching strategies: GitFlow (main + develop + feature branches — good for versioned releases), Trunk-based development (everyone commits to main frequently, short-lived branches — good for CI/CD). In DevOps you'll use both depending on the team.

Set Up SSH Key Authentication

Password SSH is insecure. Key-based SSH is what every real server uses. You'll generate a key pair on your host machine (laptop), copy the public key to the VM. The private key never leaves your laptop.

bash — Run these on your HOST machine (laptop), not the VM
# Generate an SSH key pair (run on your laptop, not the VM)
# -t ed25519 = modern, secure algorithm
# -C = comment to identify the key
ssh-keygen -t ed25519 -C "devops-lab"
# Press Enter to accept default path (~/.ssh/id_ed25519)
# Set a passphrase (optional but recommended)

# Copy your PUBLIC key to the VM
# Replace YOUR_VM_IP with the IP from the previous step
ssh-copy-id -i ~/.ssh/id_ed25519.pub devops@YOUR_VM_IP

# Now SSH in without a password!
ssh devops@YOUR_VM_IP

# If you're on Windows, use PowerShell and run:
# type $env:USERPROFILE\.ssh\id_ed25519.pub | ssh devops@YOUR_VM_IP "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
bash — Now harden SSH on the VM (run inside the VM)
# Edit SSH server config sudo vim /etc/ssh/sshd_config # Find and change these lines (remove the # to uncomment): # PermitRootLogin no ← never allow root to SSH in # PasswordAuthentication no ← keys only, no passwords # PubkeyAuthentication yes ← make sure this is yes # Restart SSH to apply changes sudo systemctl restart sshd # IMPORTANT: Don't close your existing SSH session yet! # Open a NEW terminal and test the key login works before closing this one
6

Configure the Firewall

Rocky Linux uses firewalld — a dynamic firewall manager. By default it blocks everything except SSH (port 22). You'll open ports as you need them. This is correct security practice — default deny, explicit allow.

bash — Firewall basics
# Check what's currently allowed sudo firewall-cmd --list-all # See what zone you're in (likely "public") sudo firewall-cmd --get-active-zones # Allow HTTP and HTTPS (we'll need these in Project 6) sudo firewall-cmd --permanent --add-service=http sudo firewall-cmd --permanent --add-service=https # Allow a specific port (e.g. our app will run on 3000) sudo firewall-cmd --permanent --add-port=3000/tcp # Reload to apply permanent changes sudo firewall-cmd --reload # Verify sudo firewall-cmd --list-all # You should see http, https, and 3000/tcp listed under "services" / "ports"
