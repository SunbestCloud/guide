Concepts First

AWS 101 — The Mental Model

Before writing a single line of Terraform, you need to understand how AWS thinks. These concepts are the foundation everything else builds on.

🌍 Regions, Availability Zones, and the Global Network

AWS has Regions — geographical locations (us-east-1, eu-west-1) each containing independent data centres. Within each Region are Availability Zones (AZs) — physically separate data centres connected by low-latency private fibre. Think of a Region as a city and AZs as separate neighbourhoods. A building fire in one neighbourhood doesn't affect the others.

The critical rule: always deploy across multiple AZs. A single-AZ deployment will go down. Multi-AZ deployments survive AZ failures automatically. This is why every production resource (RDS, EC2 ASG, ALB) should span at least two AZs, and ideally three.

🏙️
Analogy: A Region is like a city's power grid. AZs are separate substations. If one substation fails, the others keep running — but you need to design your buildings to draw from multiple substations. AWS gives you the substations; you have to wire your architecture correctly to use them.

🪪 IAM — The Keymaster of AWS

Every AWS API call is authenticated (who are you?) and authorised (what can you do?). IAM is the system that answers both questions. Users are people. Roles are identities that services (EC2, Lambda, CodeBuild) assume — they have no permanent credentials, only temporary ones via STS. Policies are JSON documents that grant or deny actions on resources.

The Principle of Least Privilege is not optional in AWS — it's the difference between a breach that's contained and one that compromises your entire account. Every service should have permission to do exactly what it needs, nothing more. An EC2 instance that only reads from one specific S3 bucket should have a role policy that permits only that.

🔑
Analogy: IAM is a hotel key card system. Users have their own cards. Services (like the room service robot) have temporary cards that expire after their shift. Every card is programmed to open only the specific doors needed — the robot can access the kitchen and guest floors but not the manager's office or the vault.

🌐 VPC — Your Private Cloud Network

A VPC (Virtual Private Cloud) is your own isolated network inside AWS. Think of it as the building's internal wiring — completely separate from other tenants' wiring in the same building. Within your VPC you define subnets (public-facing vs private), route tables (which traffic goes where), Internet Gateways (door to the internet), and NAT Gateways (private subnets can reach the internet to download packages, but internet can't reach them).

The standard pattern: put load balancers and bastion hosts in public subnets. Put application servers and databases in private subnets. Nothing in a private subnet should have a public IP. The NAT Gateway handles outbound internet access without exposing the instances.

📦 The AWS Shared Responsibility Model

AWS secures the infrastructure (physical hardware, network, hypervisor, managed service patching). You secure everything you put on it (your OS patches, your application security, your IAM config, your data encryption, your network rules). Many breaches happen because teams assume AWS handles more than it does.

Concrete examples: AWS patches the RDS engine for you. You must still restrict which IP ranges can reach port 5432. AWS encrypts EBS volumes at rest if you enable it. You must still rotate the access keys your app uses. The model is simple — if you configured it, you own its security.

🔢 AWS Service Categories — The Landscape

Compute: EC2 (virtual machines), Lambda (serverless functions), ECS/EKS (containers). Storage: S3 (object storage), EBS (block storage for EC2), EFS (shared filesystem). Database: RDS (managed SQL), DynamoDB (managed NoSQL), ElastiCache (managed Redis/Memcached). Networking: VPC, Route 53 (DNS), CloudFront (CDN), ALB/NLB (load balancers). Security: IAM, KMS (encryption keys), Secrets Manager, GuardDuty, WAF. Observability: CloudWatch (metrics/logs/alarms), X-Ray (distributed tracing), CloudTrail (API audit log).

Infrastructure as Code

Terraform 101 — Define Everything as Code

Terraform lets you describe infrastructure in HCL (HashiCorp Configuration Language) and have it created, updated, and destroyed automatically. If you're clicking in the AWS console, you're doing it wrong.

⚙️ How Terraform Actually Works

Terraform's workflow is always the same three steps: Write (define resources in .tf files), Plan (terraform plan — shows you exactly what will change without changing anything), Apply (terraform apply — makes the changes).

Terraform maintains a state file (terraform.tfstate) that maps your code to real AWS resources. When you run plan, Terraform compares your code to the state to determine what needs to change. Never edit the state file manually. In teams, always use a remote backend (S3 + DynamoDB) so everyone shares the same state.

🏗️
Analogy: Terraform is like an architect + construction crew combo. The .tf files are the blueprints. terraform plan is the architect reviewing the blueprints and marking what needs to be built, modified, or demolished. terraform apply is the crew executing the work. The state file is the as-built record — it documents exactly what was built so future changes can be diffed against reality.

📝 HCL Fundamentals — The Four Building Blocks

resource — declares a real infrastructure object to manage. resource "aws_instance" "web" { ... }. variable — parameterises your code. Values passed in at plan/apply time or from .tfvars files. output — exposes values after apply (e.g. the IP of the created instance). data — reads existing infrastructure Terraform didn't create (e.g. the current AWS account ID, an existing AMI, an existing VPC). These four concepts cover 90% of what you'll write.

🧱 Modules — Reusable Infrastructure Components

A module is just a directory of .tf files. Your root module is the directory you run terraform from. Child modules are subdirectories or remote sources. When your infrastructure gets large, modules let you package a VPC configuration, EC2 ASG configuration, or RDS setup as a reusable component that accepts inputs and returns outputs — exactly like a function in programming.

The pattern: keep your modules generic and parameterised (accept environment name, instance type, AZ list as variables). Use them in your root module to build environments. The same VPC module creates your dev VPC and your prod VPC with different parameters.

⚠️ Terraform Anti-Patterns to Avoid

Never use terraform apply -auto-approve in production without reviewing the plan first. A typo that removes a database will show up in the plan — catching it before apply is the whole point. Never store state locally in teams — use S3+DynamoDB backend from day one. Never hardcode account IDs, regions, or credentials in .tf files — use variables and data sources. Never import manually-created resources and call it "done" — anything in AWS that's not in Terraform will eventually cause drift and confusion.

1

Root Account Lockdown

The AWS root account has unlimited power and cannot be restricted by IAM policies. It should never be used for day-to-day operations.

  • Log in as root, go to IAM → My Security Credentials
  • Enable MFA on root using a hardware key or authenticator app
  • Delete all root access keys (there should be none)
  • Set a strong, unique root password and store in a password manager
  • Never use root again except for the 3 tasks that genuinely require it (changing support plan, closing account, enabling some services)

Enable CloudTrail — Full API Audit Log

bash — Enable CloudTrail via AWS CLI
# Create S3 bucket for CloudTrail logs
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
BUCKET="cloudtrail-logs-${ACCOUNT_ID}"

aws s3api create-bucket \
    --bucket "$BUCKET" \
    --region us-east-1

# Block all public access
aws s3api put-public-access-block \
    --bucket "$BUCKET" \
    --public-access-block-configuration "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"

# Enable bucket versioning and encryption
aws s3api put-bucket-versioning \
    --bucket "$BUCKET" \
    --versioning-configuration Status=Enabled

aws s3api put-bucket-encryption \
    --bucket "$BUCKET" \
    --server-side-encryption-configuration '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"AES256"}}]}'

# Create CloudTrail (all regions)
aws cloudtrail create-trail \
    --name myorg-cloudtrail \
    --s3-bucket-name "$BUCKET" \
    --is-multi-region-trail \
    --include-global-service-events \
    --enable-log-file-validation

aws cloudtrail start-logging --name myorg-cloudtrail

echo "✅ CloudTrail active. All API calls are now logged to s3://$BUCKET"
