Concepts First

Docker 101 — Containers Explained

Read this before writing a single Dockerfile. Understanding what's actually happening makes everything else obvious.

📦 What Is a Container? (Really)

A container is not a mini virtual machine. It's a regular Linux process with three kernel features applied: namespaces (isolation — the process can't see other processes, files, or networks outside its namespace), cgroups (resource limits — max CPU, max memory), and a layered filesystem (the container's root filesystem built from image layers).

That's it. There's no hypervisor, no separate kernel, no hardware emulation. A container shares the host's kernel. That's why containers start in milliseconds and use almost no overhead — they're just processes.

🏢
Analogy: A VM is like renting an entire building — its own structure, plumbing, electricity, everything. A container is like renting a private office inside a shared building — you have your own space and can't access other offices, but you share the building's infrastructure (kernel). Much cheaper, much faster to move in, but you can't install a different electrical system (you can't run a Windows kernel in a Linux container).

🔢 Image Layers — Why Docker Builds Are Fast

A Docker image is built from layers, stacked like transparent sheets. Each instruction in a Dockerfile (RUN, COPY, ADD) creates a new layer. Layers are content-addressed and cached — if the layer hasn't changed, Docker reuses the cached version instead of re-running the command.

This is why instruction order matters enormously. Put things that change rarely at the top (OS packages, dependencies) and things that change often at the bottom (your app code). If you copy your code before installing dependencies, every code change invalidates the dependency layer cache and reinstalls everything.

🧅
Analogy: Think of an image as an onion. Each layer is a ring. When you rebuild, Docker peels from the outside in, only redoing layers from the first changed ring outward. If you change the core (an early layer), everything outside it must be redone. Keep the stable stuff at the core.

🌐 Docker Networking — How Containers Talk

By default, each container gets its own virtual network interface. Docker creates a virtual network bridge (docker0) that containers connect to. Containers on the same user-defined network can reach each other by container name — Docker has a built-in DNS resolver. This is why your app container can connect to postgres:5432 instead of needing to know the IP.

Three key network types: bridge (default — containers on same bridge can communicate), host (container shares host network — fastest, but no isolation), none (no network — maximum isolation). In Docker Compose, every service is on the same user-defined bridge network automatically.

💾 Volumes — Data That Survives Container Death

Containers are ephemeral — when you remove one, everything written to its filesystem is gone. Volumes are the solution: storage that lives outside the container lifecycle. Three types: named volumes (managed by Docker, best for production data), bind mounts (mount a host directory into the container — perfect for development), and tmpfs mounts (in-memory, never written to disk — for sensitive data).

The rule: databases always use named volumes. Your source code in development always uses a bind mount (so changes are reflected immediately without rebuilding).

CI/CD 101 — Automate Your Entire Deploy

CI/CD is the single biggest force multiplier in DevOps. Understand it properly and you'll never want to deploy manually again.

🔄 CI vs CD — What's the Difference?

Continuous Integration (CI) means every code change is automatically built, tested, and validated. The goal: catch problems within minutes, not days. Every developer's changes are integrated into the main branch frequently (at least daily), and the pipeline proves the code works before it's merged.

Continuous Delivery (CD) means every passing build is automatically prepared for deployment. A human still clicks "deploy to production." Continuous Deployment (also CD) goes further: every passing build is deployed to production automatically with no human intervention. Most teams use Continuous Delivery — automated all the way to staging, manual gate for production.

🏭
Analogy: Think of a car factory. CI is the assembly line quality check — every car gets tested as it's built, not after 500 cars pile up. CD is the delivery truck — once a car passes QC it's automatically loaded and ready to ship. Continuous Deployment is the self-driving delivery truck — it ships itself. The value is eliminating the "pile of untested work" problem that makes big releases terrifying.

🏗️ GitHub Actions — Anatomy of a Pipeline

A GitHub Actions workflow is a YAML file in .github/workflows/. It has triggers (on push, on pull request, on schedule), jobs (parallel or sequential work units), and steps (individual commands or pre-built actions). Each job runs in a fresh container — jobs don't share state unless you explicitly pass artifacts between them.

Key concepts: Actions are reusable steps from the marketplace (e.g. actions/checkout, docker/build-push-action). Secrets are encrypted variables for API keys, never visible in logs. Environments add protection rules — require reviewers before deploying to production. Matrix builds run the same job across multiple versions in parallel.

✅ What Every Good Pipeline Must Have

1. Fast feedback: Tests should fail within 5 minutes. Run unit tests first (fast), then integration tests (slower), never the reverse. 2. Fail fast: If tests fail, stop — don't build, don't push, don't deploy. 3. Artifact immutability: Build once, promote the same artifact through environments. Never rebuild for staging that was built for dev — you'd be deploying untested code. 4. Audit trail: Every deploy is traceable to a commit, a PR, an author. 5. Rollback: You must be able to get back to the previous version in under 5 minutes.

Configuration Management

Ansible 101 — Automate Server Config

Ansible turns server setup from a manual checklist into repeatable, version-controlled, reviewable code.

🤖 What Ansible Actually Does

Ansible connects to servers over SSH and runs tasks on them. No agent required on the target server — just Python (which is on every Linux server by default) and an SSH connection. You write playbooks in YAML that describe the desired state of your servers. Ansible figures out how to get there.

The killer feature: idempotency. Run the same playbook 10 times — you get the same result. If Nginx is already installed, Ansible doesn't reinstall it. If the config file is already correct, it doesn't overwrite it. This makes Ansible safe to run repeatedly, which makes it safe to automate.

📐
Analogy: Ansible is like an architect's specification document. Instead of telling a builder "first do A, then B, then C," you specify "the result must look like this." Ansible determines what actions are needed to get from the current state to the desired state — and only takes those actions. The specification (playbook) is the source of truth, not the history of commands run.

📁 Ansible's Four Key Concepts

Inventory: The list of servers Ansible manages. Can be a static file (list of IPs/hostnames) or dynamic (queried from AWS/GCP at runtime). Servers can be in groups: [webservers], [databases], [production].

Playbook: A YAML file describing plays (which hosts get which roles/tasks). The top-level orchestrator. Role: A structured bundle of tasks, templates, variables, and handlers for one concern (e.g. a "nginx" role, a "postgresql" role). Task: A single action — install a package, copy a file, start a service, run a command.

🧩 Modules — Ansible's Vocabulary

Every task uses a module. Modules are the verbs: ansible.builtin.dnf (install packages), ansible.builtin.template (copy a Jinja2 template), ansible.builtin.service (manage services), ansible.builtin.user (manage users), ansible.builtin.copy (copy files), ansible.builtin.command / shell (run arbitrary commands — use sparingly, prefer specific modules). There are 3,000+ modules — always check if a specific module exists before using command.

Python Automation 101 — Beyond Basic Scripts

Phase 1 introduced Python. Phase 2 goes deep — AWS automation, REST API integration, Slack bots, scheduled reports, and auto-remediation systems.

🐍 Why Python for DevOps (Not Just Bash)?

Bash is perfect for orchestrating system commands. Python is better when you need: complex data structures (dicts, lists, nested objects — API responses), error handling (try/except is far richer than Bash's limited error handling), libraries (boto3 for AWS, requests for HTTP, slack_sdk for Slack — thousands of purpose-built libraries), testing (pytest makes your automation testable), and readability (complex logic is far cleaner in Python).

The practical rule: use Bash when you're gluing together CLI tools. Use Python when you're talking to APIs, processing structured data, or building something that needs to be maintained long-term.

☁️ boto3 — The AWS SDK for Python

boto3 is Amazon's official Python SDK. It lets you control every AWS service programmatically. The key concepts: client (low-level, 1:1 with the AWS API, returns raw dicts), resource (higher-level, object-oriented, nicer to use for some services), session (credentials + region configuration — use this for multi-account automation).

Every AWS console workflow you do manually can be scripted with boto3. Launching EC2 instances, creating S3 buckets, querying CloudWatch metrics, sending SNS notifications — all 200+ AWS services are accessible.

The most valuable Python automation in DevOps isn't fancy ML — it's boring, reliable operational tooling. A Slack bot that posts every deployment. A script that emails cost anomalies before your CFO sees them. A function that restarts a service when its health endpoint returns 500. An inventory script that lists every EC2 instance across all accounts. These are the scripts that make you irreplaceable on a team.

Write Your First Dockerfile

Dockerfiles are the recipe for building images. Each instruction creates a new layer.

dockerfile — Dockerfile (simple Python app)
# Base image — always pin a specific version, never use "latest" in production
FROM python:3.12-slim

# Set working directory inside the container
WORKDIR /app

# COPY requirements first — this layer is cached unless requirements change
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# NOW copy your code — this layer only invalidates when code changes
COPY . .

# Create a non-root user to run the app (security best practice)
RUN useradd --create-home appuser
USER appuser

# Document which port the app listens on (doesn't actually expose it)
EXPOSE 8000

# The command that runs when the container starts
CMD ["python", "app.py"]
bash — Build and run
# Build the image (-t = tag with a name:version)
docker build -t myapp:1.0 .

# Run it (-d = detached/background, -p = port mapping host:container)
docker run -d -p 8000:8000 --name myapp myapp:1.0

# Check it's running
docker ps
curl http://localhost:8000

# See logs
docker logs myapp
docker logs -f myapp   # follow live

# Exec into running container (for debugging)
docker exec -it myapp bash

# Stop and remove
docker stop myapp
docker rm myapp

bash — Persistent data
# Named volume — Docker manages it, survives container removal

docker run -d \
  -v $(pwd)/src:/app/src \      # your code → container

Multi-Stage Builds — Build vs Runtime

The key insight: the tools you need to build your app (compilers, build dependencies, test frameworks) are not needed to run it. Multi-stage builds let you use a fat build image and copy only the compiled output into a tiny runtime image.

dockerfile — Go app: 800MB → 12MB
# ── STAGE 1: Build ──────────────────────────────────────
FROM golang:1.22 AS builder

WORKDIR /build
COPY go.mod go.sum ./
RUN go mod download   # cache dependencies

COPY . .
# Build a statically-linked binary (no external dependencies)
RUN CGO_ENABLED=0 GOOS=linux go build -a -ldflags '-extldflags "-static"' -o myapp .

# ── STAGE 2: Runtime ────────────────────────────────────
# Scratch = completely empty image, no OS, no shell, nothing
FROM scratch

# Copy only the compiled binary from the builder stage
COPY --from=builder /build/myapp /myapp

# Copy TLS certificates (needed for HTTPS calls)
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

EXPOSE 8080
ENTRYPOINT ["/myapp"]

# Result: ~12MB image with ONLY the binary. No shell = no shell injection attacks.
dockerfile — Python app: 1.2GB → 48MB
# ── STAGE 1: Install dependencies ─────────────────────
FROM python:3.12 AS deps
WORKDIR /build
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# ── STAGE 2: Runtime ──────────────────────────────────
FROM python:3.12-slim AS runtime

# Create non-root user
RUN useradd --uid 1001 --create-home appuser

WORKDIR /app
# Copy only installed packages from build stage
COPY --from=deps /root/.local /home/appuser/.local
COPY --chown=appuser:appuser . .

USER appuser
ENV PATH=/home/appuser/.local/bin:$PATH

EXPOSE 8000
CMD ["python", "-m", "gunicorn", "app:create_app()", "-b", "0.0.0.0:8000"]
